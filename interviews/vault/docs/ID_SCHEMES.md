# Vault ID schemes

**Status:** active (revised 2026-04-25)
**Supersedes:** the 2026-04-21 `<track>-<yyyymm>-<4hex>` proposal — now retired

---

## Best-practice principles

An identifier earns its place by satisfying three properties:

1. **Stable.** Once assigned, an ID is never changed. Chain references,
   URLs, paper appendix entries, contributor bookmarks, and audit
   trails all depend on it.
2. **Collision-free at scale.** A new ID minted by any contributor or
   tool must not collide with an existing ID without a check.
3. **Human-readable enough to be quoted.** When an ID appears in a chat
   message, a PR description, or a console log, a maintainer should be
   able to make a reasonable guess about what it is.

Looking across mature systems for prior art:

| System | Pattern | Lesson |
|---|---|---|
| arXiv | `2604.12345` | Year+month+sequence — sortable, no semantic content but stable. |
| CVE | `CVE-2026-12345` | Year+sequence — stable; the registry, not the ID, conveys meaning. |
| RFC | `RFC 9999` | Pure sequence — works because the registry conveys meaning. |
| GitHub PR | `#1234` | Pure sequence — simplest possible, never confused with content. |

**Pattern: good IDs are short, stable, opaque, and presented with
semantic context elsewhere (filesystem layout, CLI, registry).** They
do *not* try to encode content in the ID itself, because content
(level, zone, topic) is mutable.

## Question ID scheme

```
<track>-<NNNN>          zero-padded, monotonic per-track sequence
```

**Examples:**

- `cloud-0001`, `cloud-2847`, `cloud-9201`
- `tinyml-0816`, `tinyml-1200`
- `edge-0972`, `mobile-0833`, `global-0306`

**Properties:**

- **Stable**: never changes after assignment.
- **Collision-free**: a `vault new` command locks `id-registry.yaml`
  to allocate the next free `<track>-NNNN`, so two contributors can't
  step on each other.
- **Sortable**: `ls cloud/` lists questions in creation order.
- **Carries the immutable axis (track)**, nothing else. Track is the
  only attribute that genuinely doesn't change.
- **Continuous with the legacy pre-2026-04-21 IDs.** The 9,000+
  existing `<track>-NNNN` IDs are preserved verbatim; new content
  continues the same numeric sequence.

### What we removed from the old proposal

The 2026-04-21 proposal was `<track>-<yyyymm>-<4hex>` (e.g.
`cloud-202604-a3f2`). We retired it for three reasons:

1. **`yyyymm` carries no useful info.** Creation month is not a search
   axis anyone uses; it's audit metadata that already lives in
   `created_at`.
2. **`4hex` content hash is unreadable.** A maintainer cannot quote
   `cloud-202604-a3f2` in a chat as easily as `cloud-2847`.
3. **It made the new IDs incompatible with the legacy IDs.** Two ID
   formats coexisting fragments tooling (sorting, search, regex).

The old scheme optimized for *deterministic regeneration* (you could
re-derive the ID from the YAML's title + topic), but in practice we
never use that property. The simpler scheme wins on every other axis.

### Pilot draft exception (transitional)

Drafts generated by automated pipelines may use a `<track>-<cohort>-<NNN>`
form (e.g., `cloud-pilot-0184`, `global-portfolio-0001`,
`mobile-balance-0007`) as a *review handle* while the question is
`status: draft`. These names trace the question back to the generation
batch for audit purposes.

**Before promotion to `status: published`**, the promotion script
(`promote_validated.py`) is responsible for assigning a durable
`<track>-NNNN` ID. The original cohort handle is preserved in
`tags:` for traceability.

## Chain ID scheme

Chains are topic-bound by definition — a chain's identity is its
topic, not its arbitrary sequence number. Encode that directly:

```
chain-<track>-<topic-slug>-<yyyymm>[-<suffix>]
```

The optional `-<suffix>` (single letter `a`, `b`, `c`, …) only appears
when two chains in the same `(track, topic, month)` bucket need to
disambiguate.

**Examples:**

- `chain-cloud-kv-cache-management-202604`
- `chain-tinyml-ota-firmware-updates-202604`
- `chain-edge-quantization-fundamentals-202605`
- `chain-cloud-latency-decomposition-202604-b` (second chain that month)

Chain IDs encode topic because chains *are* topic-bound (the schema
enforces a single topic per chain). This is different from question
IDs, which avoid encoding mutable content.

## Folder layout

Questions live under `interviews/vault/questions/<track>/<id>.yaml`.

There is a separate, deferred plan to add a second-level
`<track>/<topic>/<id>.yaml` folder structure for filesystem-level
discoverability (`ls cloud/kv-cache-management/` lists all KV-cache
questions at once). That migration is purely additive — it changes
folder paths but not IDs — so chain references and URLs survive.

When that lands, this section will be updated. As of this revision,
all questions live in the flat `<track>/` layout.

Visual sources live alongside their rendered SVG:

```
interviews/vault/visuals/<track>/<id>.svg     # the asset (always)
interviews/vault/visuals/<track>/<id>.dot     # iff source_format=dot
interviews/vault/visuals/<track>/<id>.py      # iff source_format=matplotlib
```

The renderer (`render_visuals.py`) infers source format by filename
pair, not by a YAML field — see
`interviews/vault/visuals/ARCHITECTURE.md`.

## Migration policy (no rename)

**Legacy IDs are preserved verbatim.** Every existing question ID
(`cloud-0185`, `tinyml-cell-13212`, `mobile-fill-00191`, …) stays as-is.
Same for existing chain IDs (`cloud-chain-432`, …). Renaming would
break 3,100+ chain references, the paper's appendix, the Phase 2
audit JSONLs, every external bookmark, and `git blame` chains.

**New content uses the new scheme.** `vault new` and the LLM
generator both emit IDs in the `<track>-NNNN` continuation form.

**Status today (2026-04-25):**

- Question IDs: 9,224 in the published bundle
- Chain IDs: 879 (all use the legacy `<track>-chain-<NNN>` form;
  new chain IDs adopt the topic-slug form going forward)

## Why we don't include level/zone/topic in the ID

We considered — and rejected — schemes that encoded `level` or `zone`
in the ID (e.g., `cloud-L4-kv-cache-management-0042`). Concrete reasons:

1. **Levels change.** The Phase 2 content audit (2026-04-21) moved 25
   questions across level boundaries. Every move would force an ID
   rename under a level-encoded scheme.
2. **Chain references.** ≈3,100 question references live inside
   chain YAMLs. Every rename cascades.
3. **URL rot.** The Next.js site uses IDs as anchors
   (`/practice?q=...`). ID renames break bookmarks.
4. **Git history.** Renames are traceable via `--follow`, but audit
   trails and commit messages referring to the old ID become stale.
5. **We don't need it.** The 4-axis classification is a `vault.db`
   query or a `grep ^level:` away. Tooling is cheaper than renaming.

The rule we settled on: **encode only the immutable (track), nothing
else.** Push everything else to the query layer (`vault ls
[--track --level --zone --topic --status]`).

## Companion tooling

| Need | Command |
|---|---|
| Browse all questions with filters | `vault ls [--track --level --zone --topic --status --in-chains]` |
| Inspect a single question + its chain context | `vault show <question-id>` |
| Walk a chain end-to-end | `vault chain show <chain-id>` |
| All chains for a track/topic | `vault chain ls [--track --topic]` |
| Confirm no orphan chain refs | `vault check --strict` |
| Mint a new ID at the next free slot | `vault new --track <t>` |

All read `vault.db` (compiled from YAMLs via `vault build`) or the
live YAMLs directly, so output reflects current classifications.
