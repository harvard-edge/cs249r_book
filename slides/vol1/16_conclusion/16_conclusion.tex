\begin{frame}{The System Is the Model}
\note{
% -- LINK: From objectives to the chapter's central thesis
The objectives promise synthesis. The thesis that organizes it: the
system is the model --- not just the weights file.

% -- NARRATE: Open with the mobile deployment failure

inputs, cannot be deployed reliably, or drifts without anyone noticing.
The true model is the entire system: data pipeline + training infra +
serving + monitoring.'' Point to the failure table: ``Each component has
its own failure mode. No single team owns the interaction failures.''

% -- ENGAGE: Cross-team failure experience

is a classic cross-team failure.]

\begin{frame}{Conservation of Complexity in Action}
\note{
% -- LINK: From the twelve invariants to the meta-principle that unifies them
The invariants are the laws. Conservation of Complexity is the
meta-principle: every optimization transfers complexity, never eliminates it.

% -- NARRATE: Walk through both examples in the diagram
``Quantization: you reduced bandwidth demand (Iron Law Data term shrinks)
but increased monitoring burden (Training-Serving Skew risk rises).
Abstraction: you simplified the interface but hid resource consumption
--- the complexity moved to the runtime.''
ANALOGY: ``Conservation of Complexity is to ML systems what conservation
of energy is to physics. The total never decreases.''

% -- ENGAGE: Personal experience with hidden costs
Ask: ``Can you name an optimization where the hidden cost surprised you?''
[Expected: caching (memory cost), quantization (accuracy monitoring),
model distillation (training pipeline complexity).]

% -- WARN: Students celebrate optimization gains without tracing costs
Common error: students report latency reduction without asking where
the cost went. Correct framing: every optimization is a transfer, not
an elimination. Trace the compensating cost before declaring victory.

% -- FLEX: [CORE] The meta-principle that unifies the entire course
[CORE] This is the single most important intellectual contribution.
IF AHEAD: ``Is there an optimization that truly eliminates complexity?''
IF SHORT: Show the diagram, state the rule, skip the question.
}

\begin{frame}{Conservation in Action: INT4 Quantization}
\note{
% -- LINK: From the abstract principle to a concrete worked example
The previous slide showed Conservation of Complexity with a diagram.
This slide traces a specific decision --- INT4 quantization --- to show
exactly where complexity moves.

% -- NARRATE: Walk through the three-column transfer
``Quantizing to INT4 saves 75\% memory and doubles throughput. But where
did the complexity go? Column 1: the memory savings are real --- model
shrinks from 140 GB to 35 GB. Column 2: calibration now requires
production-representative data, outlier handling, and per-channel vs
per-tensor decisions. Column 3: monitoring must now track quantization
error, detect accuracy drift from distribution shift, and maintain a
full-precision fallback.'' Point to the bottom card: ``The total
complexity did not decrease. It moved from Machine to Algorithm
(calibration) and Data (monitoring).''

\begin{frame}{Fallacies}
\note{
% -- NARRATE: Read each fallacy with its rebuttal
``Fallacy 1: Better tools eliminate complexity. Reality: Conservation of
Complexity guarantees that simplifying one interface pushes complexity to
another.'' ``Fallacy 2: Mastering components equals mastering the system.
Reality: a schema change cascades through training, breaks quantization,
and triggers silent degradation. Integration complexity exceeds the sum
of parts.'' ``Fallacy 3: A single accuracy metric captures quality.
Reality: the Pareto Frontier has seven dimensions. A 95\% model at 500 ms
may be worse than 93\% at 50 ms when P99 is the constraint.''

% -- FLEX: [CORE] Capstone fallacies synthesize the entire course
[CORE] Each fallacy references a specific invariant or principle.
IF SHORT: Cover fallacies 1 and 3, skip fallacy 2.
}

\small
\textbf{Fallacy:} \textit{Systems engineering complexity disappears with better tools.}\\
{\footnotesize Tools abstract complexity; they do not eliminate it. The Conservation of Complexity guarantees that simplifying one interface pushes complexity to another.}

\vspace{0.15cm}
\textbf{Fallacy:} \textit{Mastering individual components equals mastering the system.}\\
{\footnotesize A data schema change cascades through training, breaks quantization assumptions, and triggers silent accuracy degradation. Integration complexity exceeds the sum of component complexities.}

\vspace{0.15cm}
\textbf{Fallacy:} \textit{A single accuracy metric captures model quality.}\\
{\footnotesize The Pareto Frontier: accuracy, latency, throughput, memory, energy, fairness, cost. A 95\% acc model at 500 ms may be worse than 93\% at 50 ms when P99 is the hard constraint.}

\end{frame}

\begin{frame}{Pitfalls}
\note{
% -- NARRATE: Read each pitfall with its quantitative evidence
``Pitfall 1: Optimizing one invariant while ignoring Conservation of
Complexity. Quantization shifts load to accuracy monitoring. Caching
trades memory for speed. Always ask: where did the cost go?'' ``Pitfall
2: Optimizing one stage without profiling the full system. Amdahl's Law:
10x inference speedup yields only 1.1x system speedup if data loading is
90\% of latency. The Iron Law identifies the dominant term first.''

% -- FLEX: [CORE] Capstone pitfalls --- both share a common root
[CORE] The temptation to reduce a system to its parts.
IF SHORT: Cover pitfall 2 (more quantitative), skip pitfall 1.
}

\small
\textbf{Pitfall:} \textit{Optimizing a single invariant while ignoring the Conservation of Complexity.}\\
{\footnotesize When an optimization reduces latency by 50\%, ask where the cost went. Quantization shifts load to accuracy monitoring. Caching trades memory capacity for speed. Engineers who celebrate gains in one metric without tracing compensating costs build systems that fail in unexpected ways.}

\vspace{0.2cm}
\textbf{Pitfall:} \textit{Optimizing a single pipeline stage without profiling the full system.}\\
{\footnotesize Amdahl's Law: optimizing inference by 10$\times$ yields only 1.1$\times$ system speedup if data loading accounts for 90\% of latency. The Iron Law decomposes execution time precisely so engineers can identify the dominant term \emph{before} investing effort.}

\end{frame}
