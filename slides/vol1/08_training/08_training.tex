\begin{frame}{Why Training Costs Millions}
\note{
% -- LINK: What prior concept connects to this slide
Ch7 showed how frameworks execute individual forward/backward passes.
This slide reveals the scale: training requires \emph{billions} of those
passes, creating a million-to-one cost asymmetry between inference and
training.

% -- NARRATE: What to SAY while showing this slide
``Running a model costs pennies. Training a model costs millions. GPT-2
inference: fraction of a cent. GPT-2 training: \$50K. GPT-4 training:
an estimated \$100M. This million-to-one gap governs who can participate
in AI research and deployment.''
Point to the three scaling challenges: ``Sustained petaFLOPS for days
(compute). Weights plus gradients plus optimizer plus activations (memory).
Sequential gradient updates that cannot be trivially parallelized
(data dependencies).''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``Who can afford to train a frontier model?'' Let 2 students
respond. [Expected answer: only large companies with datacenter-scale
budgets. This motivates optimization.]

price. Correct framing: electricity, cooling, and engineering time
often exceed hardware cost for large runs. A 10-day run on 1000 GPUs
at \$2/GPU-hr costs \$480K in compute alone.

\begin{frame}{The 3AM Gradient Explosion}
\note{
% -- LINK: What prior concept connects to this slide
Training costs millions. Losing a run to a numerical failure wastes
those resources entirely. This story motivates the safety techniques
covered later (checkpointing, gradient clipping, BF16).

% -- NARRATE: What to SAY while showing this slide
Tell this as a story: ``A team trains a 7B model for 4 days. Everything
looks smooth. At 3AM, an outlier batch produces a gradient that exceeds
FP16's maximum value of 65,504. The gradient overflows to infinity.
Infinity times any weight equals NaN. Within one step, every parameter
is NaN. The run crashes. Four days and \$5,000 lost.''
Point to the systems fix: ``Three defenses. Checkpointing: save hourly,
lose 1 hour instead of 4 days. Gradient clipping: cap gradient norms
before they overflow. BF16: matches FP32's dynamic range in 16 bits,
making overflow extremely rare.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``Why does BF16 solve the overflow problem while FP16 does not?''
Give 10 seconds. [Expected answer: BF16 has the same 8-bit exponent as
FP32 (range up to $10^{38}$), while FP16 has only a 5-bit exponent
(max 65,504).]

% -- WARN: What students will get wrong on THIS topic
Common error: students think gradient explosions are rare edge cases.
Correct framing: at scale (billions of tokens), outlier batches are
statistically guaranteed. Without safeguards, every long run will
eventually hit one.

% -- FLEX: [CORE] Motivates checkpointing, clipping, and BF16 covered later.
[CORE] The FP16/BF16 comparison is referenced in the mixed-precision section.
IF SHORT: Tell the story in 60 seconds; skip the table, keep the three fixes.
}

\scriptsize
\begin{columns}[T]
  \begin{column}{0.55\textwidth}
    \textbf{Scenario:} Training a 7B LLM. Loss smooth for 4 days.\\
    \textbf{3:00 AM:} Loss spikes to \textbf{NaN}. Crash. Lost 4 days (\$5K).

    \vspace{0.05cm}
    \textbf{Physics:} Outlier batch exceeded FP16 range ($>$65,504).\\
    Gradients $\to$ Inf $\to$ all weights NaN.

    \vspace{0.05cm}
    \textbf{Systems Fix:}
    \begin{itemize}\setlength\itemsep{0pt}
      \item \textbf{Checkpointing}: Save hourly (lose 1hr, not 4 days)
      \item \textbf{Gradient clipping}: Cap gradient norms
      \item \textbf{BF16}: FP32 range in 16 bits; overflows rare
    \end{itemize}

\section{Iron Law}

\mlsysfocus{The Iron Law of Training Performance}{%
$T_{\text{train}} = \dfrac{O}{R_{\text{peak}} \times \eta}$%
}

\begin{frame}{Iron Law: Three Levers}
\note{
% -- LINK: What prior concept connects to this slide
The focus frame just presented the Iron Law equation. This slide unpacks
the three terms and shows how each maps to a class of optimizations.

% -- NARRATE: What to SAY while showing this slide
Walk through each term in the diagram: ``$O$ = total FLOPs for the
training run. Set by model size, dataset size, and number of epochs.
$R_{\text{peak}}$ = hardware throughput---set by GPU procurement and
numerical precision. $\eta$ = utilization---the fraction of peak
throughput actually achieved. This is the engineering target.''
``GPT-3 achieved $\eta = 0.45$. Current target for well-optimized runs:
above 0.55. That gap is where optimization lives.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``Which of the three levers is cheapest to improve?'' Give 15
seconds. [Expected answer: $\eta$---utilization is pure engineering.
$O$ requires algorithmic changes, $R_{\text{peak}}$ requires hardware
purchases.]

% -- WARN: What students will get wrong on THIS topic
Common error: students focus on $R_{\text{peak}}$ (buy faster GPUs)
instead of $\eta$ (use existing GPUs better). Correct framing: most
training runs achieve 30--50\% utilization. Doubling $\eta$ from 0.3
to 0.6 is equivalent to buying 2x more hardware.

% -- FLEX: [CORE] The Iron Law is the organizing framework for the entire chapter.
[CORE] Every optimization slide maps back to one of these three terms.
IF SHORT: Name the three terms and the GPT-3 utilization number; skip
the procurement discussion.
}

\begin{frame}{Iron Law: Optimization Mapping}
\note{
% -- LINK: What prior concept connects to this slide
The previous slide defined the three levers. This table shows which
optimization targets which lever---the organizing framework for the
rest of the deck.

% -- NARRATE: What to SAY while showing this slide
Point to each row: ``Mixed precision increases $R_{\text{peak}}$---Tensor
Cores at 16x higher FLOP/s. Data prefetching increases $\eta$---reduces
idle time. Gradient checkpointing increases $O$ (adds recomputation) but
enables larger models. FlashAttention both reduces $O$ and improves
$\eta$ through IO-aware tiling.''
``The diagnostic discipline: identify which term dominates \emph{before}
choosing an optimization. Do not apply mixed precision if your bottleneck
is data loading.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``If your GPU profiler shows 40\% idle time waiting for data, which
column should you target?'' Give 10 seconds.
[Expected answer: $\eta$---data prefetching to eliminate idle time.]

% -- WARN: What students will get wrong on THIS topic
Common error: students apply optimizations in random order without
diagnosing the bottleneck first.
Correct framing: always profile first. If the bottleneck is data loading
($\eta$), mixed precision ($R_{\text{peak}}$) will not help.

% -- FLEX: [CORE] This table is the roadmap for all remaining slides.
[CORE] Students should be able to classify any optimization by its term.
IF SHORT: Cover mixed precision and data prefetching rows; skip the rest.
}

\begin{frame}{Training vs.\ Inference: The 4$\times$ Multiplier}
\note{
% -- LINK: What prior concept connects to this slide
Students just predicted training memory. This slide reveals the answer:
the 4x multiplier from optimizer state, plus activations on top.

% -- NARRATE: What to SAY while showing this slide
``Reveal time. Weights: 14 GB. Gradients: another 14 GB. Adam first
moment ($m$): 14 GB. Adam second moment ($v$): 14 GB. That is 56 GB
minimum---4x the weight memory. And we have not counted activations yet.
GPT-2 XL in FP32: 89 GB total. A V100 has 32 GB. This is why
optimization matters---a model that runs inference on one GPU needs
multiple GPUs for training.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Compare students' earlier predictions to the actual 56+ GB number.
Ask: ``How many of you predicted less than 30 GB?'' [Most hands.]
``That gap between expectation and reality is why this chapter exists.''

% -- WARN: What students will get wrong on THIS topic
Common error: students think weights dominate training memory.
Correct framing: optimizer state (Adam's $m$ and $v$) doubles the
non-activation memory. Activations can add another 3--4x on top.
Weights are often the \emph{smallest} component.

% -- FLEX: [CORE] The 4x multiplier is the most important number in this chapter.
[CORE] Referenced in every subsequent optimization slide.
IF AHEAD: ``What if we used SGD instead of Adam? How would the multiplier
change?'' [Answer: 2x instead of 4x.]
}

\begin{frame}{Optimizer Memory: SGD vs.\ Adam}
\note{
% -- LINK: What prior concept connects to this slide
The 4x multiplier comes from Adam's two extra state vectors per parameter.
This slide compares optimizer memory costs to explain that multiplier.

% -- NARRATE: What to SAY while showing this slide
Point to the table: ``SGD: weights + gradients = 2x memory. SGD with
momentum: add one momentum buffer = 3x. Adam: add two buffers ($m$ and
$v$) = 4x. For a 7B model in mixed precision: SGD needs 28 GB, Adam
needs 56 GB.''
``Adam converges in roughly 1/3 the iterations of SGD, but costs 3x
the memory for per-parameter state. This is a binding constraint for
large models.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``When would you choose SGD over Adam?'' Give 15 seconds.
[Expected answer: when memory is the binding constraint---e.g., a very
large model that barely fits with SGD but overflows with Adam. Accept
slower convergence for the ability to train at all.]

% -- WARN: What students will get wrong on THIS topic
Common error: students always default to Adam because it converges faster.
Correct framing: Adam's faster convergence is irrelevant if the model
does not fit in memory. Memory is a hard constraint; convergence speed
is a soft preference.

% -- FLEX: [CORE] Explains the 4x multiplier and the memory-convergence trade-off.
[CORE] Students must know the memory cost of each optimizer for the exercise.
IF SHORT: Cover the table and the 4x number; skip AdamW detail.
}

\begin{frame}{Activation Memory: The Hidden Cost}
\note{
% -- LINK: What prior concept connects to this slide
Optimizer state gives the 4x multiplier. Activations add even more---
often exceeding optimizer state for deep models.

% -- NARRATE: What to SAY while showing this slide
``Activations must be stored from the forward pass for use in the
backward pass (the autograd tape from Ch7). Memory scales with batch
size times sequence length times hidden dimension times layers.
GPT-2 XL: 48 layers, batch 32---activations alone are 65 GB. That
is 73\% of total training memory. Activations, not weights, are the
dominant cost.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``If you halve the batch size, how does activation memory change?''
Give 10 seconds. [Expected answer: it halves---activation memory is
linear in batch size.]

% -- WARN: What students will get wrong on THIS topic
Common error: students think reducing model parameters is the primary
way to reduce training memory.
Correct framing: activations scale with batch size and depth, not just
parameters. Reducing batch size or applying checkpointing are often
more effective than shrinking the model.

% -- FLEX: [CORE] Activations as 73\% of memory motivates checkpointing later.
[CORE] The GPT-2 breakdown is the key quantitative reference.
IF SHORT: Show the formula and the 73\% number; skip the full table.
}

\begin{frame}{Mixed-Precision Training}
\note{
% -- LINK: What prior concept connects to this slide

\begin{frame}{Gradient Accumulation}
\note{
% -- LINK: What prior concept connects to this slide
The memory budget exercise showed that large batch sizes cause OOM.
Gradient accumulation solves this by splitting the effective batch into
smaller micro-batches that fit in memory.

% -- NARRATE: What to SAY while showing this slide
``Problem: batch size 256 gives best convergence but requires too much
activation memory. Solution: split into 8 micro-batches of 32. Run
forward/backward on each micro-batch, accumulate gradients without
updating, then do one optimizer step with the averaged gradients.
Physical batch = 32 (fits in memory). Effective batch = 256 (good
convergence). Overhead: 5--10\% from multiple forward/backward passes
per update step.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``Is gradient accumulation mathematically equivalent to training
with the full batch?'' Give 10 seconds.
[Expected answer: yes for SGD. For Adam, nearly equivalent---the running
statistics $m$ and $v$ see the same averaged gradient. The only difference
is batch normalization statistics, which see the micro-batch.]

% -- WARN: What students will get wrong on THIS topic
Common error: students think gradient accumulation is free.
Correct framing: each micro-batch requires a separate forward and backward
pass. With 8 micro-batches, you do 8x the kernel launches per update,
adding 5--10\% overhead from dispatch and synchronization.

[CORE] Completes the optimization toolkit alongside mixed precision and

\begin{frame}{Data Parallelism: Replicate Model, Split Data}
\note{
% -- LINK: What prior concept connects to this slide
Single-machine optimizations have a ceiling: one GPU has finite memory
and compute. Data parallelism is the simplest multi-GPU strategy---
replicate the model, split the data.

% -- NARRATE: What to SAY while showing this slide
Walk through the diagram: ``Each GPU holds a full copy of the model.
The mini-batch splits across GPUs---each processes its shard independently.
After the backward pass, AllReduce averages the gradients across all
GPUs. Communication cost: approximately 2x model size per step.
NVLink with 4 GPUs gives ~3.6x speedup. PCIe: only ~2.5x because of
lower bandwidth.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``Why doesn't 4 GPUs give exactly 4x speedup?'' Give 15 seconds.
[Expected answer: AllReduce communication takes time. The gradient sync
step is pure overhead that does not exist in single-GPU training.]

% -- WARN: What students will get wrong on THIS topic

Correct framing: communication overhead grows with model size and number
of GPUs. The speedup is always sub-linear: $S(N) < N$ because of the
AllReduce step.

[CORE] The sub-linear scaling motivates the ``exhaust single-machine first'' rule.

\begin{frame}{The Communication Tax}

\begin{frame}{Fallacies}
\note{
% -- LINK: What prior concept connects to this slide
Students applied Iron Law diagnostics in the discussion exercise.
These fallacies show common reasoning errors in training optimization.

% -- NARRATE: What to SAY while showing this slide
Read each fallacy: ``Fallacy 1: larger models always perform better.
A 20B model trained on insufficient data underperforms a 7B model by
3--5\% due to overfitting. Model capacity must match dataset size.''
``Fallacy 2: hyperparameters transfer across scales. Learning rate at
batch 32 does not work at batch 4096---you must apply the linear scaling
rule. Ignoring this causes divergence 2--5 days into a multi-week run.''
``Fallacy 3: mixed precision is a simple toggle. FP16 without loss
scaling causes gradient underflow. A 72-hour run can diverge at step
40K due to accumulated numerical errors.''

% -- FLEX: [CORE] Fallacies reinforce quantitative reasoning about training.
[CORE] Each maps to a specific Iron Law concept.
IF SHORT: Cover fallacies 1 and 3; skip fallacy 2.
}

\small
\textbf{Fallacy:} \textit{Larger models always yield better performance.}\\
{\footnotesize A 20B model trained on insufficient data ($<$500M examples) \emph{underperforms} a 7B model by 3--5\% due to overfitting. Model capacity must match dataset size.}

\vspace{0.15cm}
\textbf{Fallacy:} \textit{Hyperparameters transfer from small-scale to large-scale training.}\\
{\footnotesize Learning rate at batch 32 does not work at batch 4096. Linear scaling rule: multiply LR by the batch ratio. Ignoring this causes divergence 2--5 days into a multi-week run.}

\vspace{0.15cm}
\textbf{Fallacy:} \textit{Mixed precision is a simple toggle.}\\
{\footnotesize FP16 without loss scaling causes gradient underflow. A model training for 72 hours can diverge at step 40K due to accumulated numerical errors. Always validate on representative workloads.}

\end{frame}

\begin{frame}{Pitfalls}
\note{
% -- LINK: What prior concept connects to this slide
Fallacies are wrong beliefs; pitfalls are operational mistakes.
These three map to scaling, optimization independence, and data pipeline.

% -- NARRATE: What to SAY while showing this slide

models on 8 GPUs spend 30--50\% synchronizing. A single well-optimized

accumulation trades 5--10\% efficiency for 4x memory reduction.''
``Pitfall 3: neglecting data pipeline. Data loading creates 20--40\%

assuming the GPU is the bottleneck.''

\begin{frame}{Key Takeaways}
\note{
% -- LINK: What prior concept connects to this slide
Students just did retrieval practice. This slide reveals the answers.

% -- NARRATE: What to SAY while showing this slide
Walk through each bullet: ``Iron Law: $T = O / (R \times \eta)$.
Every optimization targets one term.''
``4x memory multiplier: Adam adds 3 extra copies. Activations scale
with batch times depth.''
``Mixed precision: 2x throughput, 2x memory, <1\% accuracy. BF16
eliminates loss scaling.''
``FlashAttention: IO-aware tiling, 2--4x speedup, same exact math.''
``Gradient checkpointing: 10x less memory for 33\% more compute.''
``Profile before optimize: find the bottleneck first.''
``Scaling: communication overhead makes multi-GPU sub-linear. Exhaust
single-machine first.''

% -- FLEX: [CORE] Consolidates the lecture.
[CORE] Read all bullets; emphasize the Iron Law as the organizing framework.
IF SHORT: Read bullets 1, 2, and 6; let students read the rest.
}

\scriptsize
\begin{itemize}\setlength\itemsep{0pt}
  \item \textbf{Iron Law}: $T_{\text{train}} = O / (R_{\text{peak}} \times \eta)$. Every optimization targets one of these three terms.
  \item \textbf{4$\times$ memory multiplier}: Adam adds 3 extra copies per parameter. Activations scale with batch $\times$ depth.
  \item \textbf{Mixed precision}: 2$\times$ throughput, 2$\times$ memory savings, $<$1\% accuracy impact. BF16 eliminates loss scaling.
  \item \textbf{FlashAttention}: IO-aware tiling converts attention from memory-bound to compute-bound. 2--4$\times$ speedup.
  \item \textbf{Gradient checkpointing}: 10$\times$ less activation memory for 33\% more compute.
  \item \textbf{Profile before optimize}: Diagnose the bottleneck (compute, memory, or data) before choosing a technique.
  \item \textbf{Scaling trade-off}: Multi-GPU adds communication overhead. Exhaust single-machine optimizations first.
\end{itemize}

\end{frame}

  \end{column}
\end{columns}

\centering
{\normalsize Training taught us \emph{how} to learn efficiently.}\\[0.1cm]
{\small\textbf{Data Selection asks: what should the model learn \emph{from}?}}

\end{frame}
