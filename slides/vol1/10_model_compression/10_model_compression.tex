\begin{frame}{The Physics of Quantization}
\note{
% -- LINK: The deployment gap showed a size problem; now reveal the energy problem
The previous slide showed models do not fit in RAM. This slide shows why
even fitting is not enough --- moving data costs orders of magnitude more
than computing on it.

% -- NARRATE: Walk through the energy table for the ``aha'' moment
Point to the table: ``An 8-bit integer add costs 1x energy. A 32-bit float
add costs 30x. But a DRAM read costs 40,000x. The bottleneck is not
arithmetic --- it is data movement.'' Then point to the blue card: quantizing
from FP16 to INT4 gives 4x speedup because the model is bandwidth-bound.

% -- ENGAGE: Force students to identify the optimization target
Ask: ``If memory access costs 40,000x more than an integer add, where
should we focus optimization?'' Give 20 seconds. Expected: reduce data
movement, not reduce compute.

% -- WARN: Students fixate on compute cost, not data movement cost
Common error: students assume faster ALUs are the answer because GPUs
are marketed on TFLOPS. Correct framing: quantization wins because it
reduces bytes moved, not operations performed.

% -- FLEX: [CORE] The energy hierarchy is the physical basis for all compression
[CORE] This slide is essential --- it grounds every technique in physics.
IF AHEAD: ``What does this imply for the relative value of pruning vs quantization?''
IF SHORT: Skip the question, keep the table walkthrough.
}

\section{Optimization Stack}
% =============================================================================

\begin{frame}{The Three-Layer Optimization Stack}
\note{
% -- LINK: Energy hierarchy showed why compression matters; now show the full toolkit
The energy slide proved data movement dominates. This slide introduces the
three-layer stack that attacks the problem from three angles simultaneously.

% -- NARRATE: Walk through each layer on the diagram
Point to top layer: ``Structural --- what to compute (pruning, distillation, NAS).''
Middle: ``Precision --- how precisely (FP32 to INT8 to INT4).''
Bottom: ``Architectural --- how efficiently (fusion, sparsity).''
Then the punchline: ``They compound multiplicatively: 2x times 4x times 2x = 16x.''

% -- ENGAGE: Test multiplicative vs additive intuition
Ask: ``If pruning gives 2x and quantization gives 4x, what is the combined
speedup?'' Students often say 6x (additive). Correct: 8x minimum, up to 16x
with fusion.

% -- WARN: Students assume techniques are additive, not multiplicative
Common error: adding compression ratios (2+4+2=8x).
Correct framing: each layer reduces a different bottleneck independently,
so gains multiply when layers target orthogonal constraints.

% -- FLEX: [CORE] This is the chapter's organizing framework
[CORE] Every subsequent slide maps to one of these three layers.
IF AHEAD: ``Which layer has the highest risk of accuracy loss?''
IF SHORT: Show the diagram, state the 16x result, move on.
}

\note{

\section{Pruning}
% =============================================================================

\begin{frame}{Pruning: Remove What Doesn't Matter}
\note{
% -- LINK: The optimization stack's top layer is structural --- pruning is its primary tool
The three-layer stack's top layer addresses ``what to compute.'' Pruning
removes weights that carry minimal information, directly reducing model size.

% -- NARRATE: Walk through the formal objective, then the heuristic
Point to the equation: ``We want the smallest set of weights that preserves
accuracy. This is NP-hard, so we use magnitude pruning: if a weight is
below threshold tau, zero it out.'' Then: ``ResNet-50 has 25.6M params but
only 3M carry task information --- 90\% can be pruned with under 2\% loss.''

% -- ENGAGE: Ask why networks are overparameterized in the first place
Ask: ``Why are networks overparameterized?'' Give 20 seconds.
Expected: overparameterization helps optimization (loss landscape smoothing),
but the final solution is sparse.

% -- WARN: Students assume pruning automatically speeds up inference
Common error: ``I pruned 90\% of weights so inference is 10x faster.''
Correct framing: sparse matrices need hardware support to skip zeros.
Without sparse kernels, the pruned model runs at the same speed.

% -- FLEX: [CORE] Pruning is a fundamental technique referenced throughout
[CORE] Essential for the structured-vs-unstructured distinction next.
IF AHEAD: ``What is the relationship between pruning and the Lottery Ticket Hypothesis?''
IF SHORT: Skip the audience question, walk through the equation and numbers.
}

\footnotesize
\begin{columns}[T]
  \begin{column}{0.52\textwidth}
    Networks carry far more parameters than any task requires.

    \vspace{0.05cm}
    \begin{mlsyscard}{crimson}
    {\scriptsize \textbf{Pruning} removes weights that contribute minimal information.}
    \end{mlsyscard}

    \vspace{0.05cm}
    \textbf{Formal objective:}
    $\min_{\hat{W}} \mathcal{L}(\hat{W}) \;\; \text{s.t.} \;\; \|\hat{W}\|_0 \leq k$

    \vspace{0.05cm}
    {\scriptsize
    NP-hard $\to$ magnitude heuristic:
    $M_{i,j} = \begin{cases} 1 & |W_{i,j}| > \tau \\ 0 & \text{otherwise} \end{cases}$
    }
  \end{column}
  \begin{column}{0.45\textwidth}
    {\scriptsize
    \textbf{MobileNet:} 14 MB $\to$ 2 MB, $<$1\% loss\\
    \textbf{ResNet-50:} 90\% prunable (3M of 25.6M params carry task information)
    }

    \vspace{0.1cm}
    \mlsysalert{Pitfall:}{Pruning does \emph{not} auto-speed inference --- sparse matrices need HW support.}
  \end{column}
\end{columns}

\end{frame}

\begin{frame}{Structured vs.\ Unstructured Pruning}
\note{
% -- LINK: Pruning was introduced abstractly; now show it has two fundamentally different forms
The previous slide warned that pruning does not auto-speed inference.
This slide explains why: it depends on the pruning structure.

% -- NARRATE: Walk through the three columns in the diagram
Point left: ``Unstructured --- scattered zeros everywhere. Hardware cannot skip
individual zeros efficiently. You need sparse matrix kernels.''
Point center: ``N:M structured --- NVIDIA's 2:4 pattern. Exactly 2 of every 4
weights are zero. A100 Sparse Tensor Cores give 2x speedup.''
Point right: ``Channel/layer pruning --- remove entire columns. The result is
a smaller dense network that runs on any hardware.''

% -- ENGAGE: Which type would you choose for a mobile phone?
Ask: ``Which pruning type would you choose for a phone with no sparse kernel
support?'' Expected: channel pruning (dense sub-network, any hardware).

% -- WARN: Students think all pruning gives speedup
Common error: ``90\% pruned = 10x faster'' regardless of structure.
Correct framing: unstructured pruning saves memory but not latency unless
the hardware has sparse execution support.

% -- FLEX: [CORE] This distinction determines real-world deployment decisions
[CORE] The structured-vs-unstructured choice is a recurring design decision.
IF AHEAD: ``Why did NVIDIA choose 2:4 specifically?'' (hardware-friendly pattern)
IF SHORT: Just cover the three columns, skip the engagement question.
}

\begin{frame}{Iterative vs.\ One-Shot Pruning}
\note{
% -- LINK: Structured vs unstructured was about where to prune; this is about how much at once
The previous slide showed pruning types. This slide shows pruning schedules:
gradual removal with recovery vs aggressive one-shot removal.

% -- NARRATE: Walk through the table, then contrast with one-shot
Point to the table: ``Each cycle prunes 2 channels, accuracy drops, fine-tuning
recovers it. After 3 cycles: 27\% channels removed, only 0.4\% accuracy lost.''
Then: ``One-shot at 27\%: accuracy drops to 0.943 --- a 5\% degradation.''
Point to the Lottery Ticket card: ``Dense networks contain sparse subnetworks
that match full accuracy from the original initialization.''

% -- ENGAGE: Ask why iterative works so much better
Ask: ``Why does removing 2 channels at a time work better than removing all
at once?'' Expected: the network can redistribute information during fine-tuning.

% -- WARN: Students underestimate the cost of iterative pruning
Common error: thinking iterative pruning is always worth the extra training.
Correct framing: iterative requires N fine-tuning rounds. For large models
(LLMs), this can be prohibitively expensive. One-shot + calibration is
sometimes the only practical option.

\section{Distillation}
% =============================================================================

\begin{frame}{Knowledge Distillation: Learning from a Teacher}
\note{
% -- LINK: Pruning removes weights from an existing model; distillation trains a new smaller one
Pruning carved away parts of an existing network. Distillation takes a
different approach: train a compact student from scratch using the teacher's
soft probability outputs.

% -- NARRATE: Walk through the pipeline diagram, then the loss equation
Point to the diagram: ``Teacher produces soft labels at temperature T.
Student learns from both the soft labels (KL divergence) and the hard labels
(cross-entropy). Alpha balances the two losses.'' Then point to DistilBERT
card: ``97\% of teacher accuracy, 40\% fewer parameters, 60\% faster.''
ANALOGY: ``The teacher writes a detailed answer key with partial credit ---
the student learns not just what is right, but what is almost right.''

% -- ENGAGE: Why do soft labels help more than hard labels?
Ask: ``Why do soft labels help?'' Give 20 seconds. Expected: soft labels
reveal inter-class similarity (e.g., a ``7'' looks somewhat like a ``1'').
Hard labels hide this structure.

% -- WARN: Students confuse temperature with learning rate
Common error: thinking higher T always means better distillation.
Correct framing: T controls the entropy of the output distribution.
Too high: everything looks uniform. Too low: no dark knowledge transfer.
Typical values: T = 3--20.

% -- FLEX: [CORE] Distillation is a distinct paradigm from pruning
[CORE] Students need this to understand the comparison table next.
IF AHEAD: ``Can you distill a pruned model into an even smaller student?''
IF SHORT: Skip the analogy, keep the diagram walkthrough and DistilBERT numbers.
}

\begin{frame}{Distillation vs.\ Pruning}
\note{
% -- LINK: Students now know both pruning and distillation; compare them head-to-head
The previous slides introduced pruning (carve existing model) and distillation
(train a new smaller model). This table makes the trade-off explicit.

% -- NARRATE: Walk through the comparison table row by row
Point to Mechanism: ``Pruning subtracts from an existing model. Distillation
builds from scratch.'' Output: ``Pruning can produce sparse matrices needing
special kernels. Distillation always produces dense models.'' Then the
Conservation of Complexity card: ``Neither technique destroys complexity ---
pruning pushes it to hardware, distillation pushes it to training budget.''

% -- ENGAGE: Which would you choose for a resource-constrained team?
Ask: ``If your team has limited GPU budget for retraining, which do you pick?''
Expected: pruning (PTQ-style, no retraining). If unlimited training budget:
distillation (better final result).

% -- WARN: Students think one technique is strictly better than the other

\begin{frame}{How Quantization Works}
\note{
% -- LINK: The focus frame just stated the 4x rule; now show the mechanism
The dark-background slide stated FP32-to-INT8 gives 4x memory. This slide
reveals how: snapping continuous values to a discrete grid.

% -- NARRATE: Walk through the number line diagram left to right
Point to FP32 (continuous line): ``Every real number is representable.''
Point to INT8 (256 grid points): ``Only 256 discrete levels. The quantization
error is the gap between the true value and the nearest grid point.''
Then: ``The formula: x\_q = round(x / scale) + zero\_point. This is lossy ---
you cannot recover the original value.''
Point to the Free Lunch Zone text: ``FP32 to INT8 loses less than 1\%.
Below 4 bits: the Quantization Cliff --- accuracy collapses.''

% -- ENGAGE: What happens when distinct values collide?
Ask: ``What happens when two very different FP32 values map to the same
INT8 level?'' Give 20 seconds. Expected: information is permanently lost;
the model cannot distinguish those two activations.

% -- WARN: Students think quantization is rounding and therefore harmless
Common error: ``it is just rounding, so the error is small.''
Correct framing: outlier activations in transformers can be 100x larger
than typical values. Uniform quantization wastes most of its 256 levels
on the narrow central range, crushing outliers.

% -- FLEX: [CORE] This is the conceptual foundation for PTQ vs QAT next
[CORE] Students must understand the lossy mapping before comparing strategies.
IF AHEAD: ``How would you handle outlier activations that are 100x the mean?''
IF SHORT: Skip the question, keep the diagram walkthrough.
}

\begin{frame}{PTQ vs.\ QAT: When Precision Matters}
\note{
% -- LINK: The number line showed quantization is lossy; now show two strategies to manage the loss
Students understand quantization introduces error. This slide presents
two strategies: accept the error (PTQ) or train through it (QAT).

% -- NARRATE: Walk through PTQ first, then QAT diagram
Left column: ``PTQ: quantize the trained model and deploy. No retraining.
This is the strong baseline --- always try PTQ first.''

percentage points when PTQ is not enough.''

\begin{frame}{Fallacies}
\note{
% -- LINK: Students just designed pipelines composing techniques; now show how composition fails
The discussion exercise assumed techniques compose cleanly. These fallacies
show where that assumption breaks down.

% -- NARRATE: Walk through each fallacy with its quantitative rebuttal
First fallacy: ``Pruning to 70\% gives 97.8\%. Adding INT8 should still be
97\%, right? No --- drops to 94.2\%. Techniques interact non-linearly.''
Second: ``INT8 loses 0.1\%. INT4 should lose 0.2\%? No --- 5--15\% drop.
Binary: 51\%. The curve has a cliff, not a slope.''
Third: ``INT8 + 50\% pruning should give 8x speedup. Actual: 28\%.
Sparse dequantization overhead erodes the theoretical gains.''

% -- WARN: The interaction fallacy is the most dangerous for pipeline design
Common error: multiplying individual compression ratios to predict combined
speedup. Correct framing: always measure the combined pipeline end-to-end.
Theoretical composition overestimates real gains by 2--5x.

% -- FLEX: [CORE] Fallacies directly challenge what students learned earlier
[CORE] These fallacies are essential correctives to the optimistic framing.
IF AHEAD: ``How would you test for non-linear interactions in your pipeline?''
IF SHORT: Cover just the first fallacy (interaction), skip the other two.
}

\footnotesize
\textbf{Fallacy:} \textit{Optimization techniques compose independently.}\\
{\scriptsize BERT pruned to 70\%: 97.8\% accuracy. Add INT8: drops to 94.2\%. Techniques interact non-linearly.}

\vspace{0.1cm}
\textbf{Fallacy:} \textit{Aggressive quantization scales uniformly.}\\
{\scriptsize ResNet-50 INT8: 76.1\% (vs.\ 76.2\% FP32). INT4: 5--15\% drop. Binary: \textbf{51\%}. Threshold effects.}

\vspace{0.1cm}
\textbf{Fallacy:} \textit{Compression ratios predict deployment speedup.}\\
{\scriptsize INT8 + 50\% pruning $\to$ expected 8$\times$. Actual: \textbf{28\%} end-to-end. Sparsity + dequantization overhead erode gains.}

\end{frame}

\begin{frame}{Pitfalls}
\note{
% -- LINK: Fallacies showed conceptual errors; pitfalls show operational mistakes
The fallacies were about wrong mental models. These pitfalls are about
wrong engineering practices --- optimizing the wrong metric and skipping
training-aware methods.

% -- NARRATE: Walk through each pitfall with its numbers
First: ``A team reduces FLOPs by 60\% but achieves only 12\% latency reduction
on ARM. Why? Sparse patterns prevent vectorization. Memory bandwidth and
cache behavior determine actual performance --- not operation counts.
The rule: profile on target hardware.''
Second: ``PTQ achieves 96.8\% of BERT baseline. QAT retains 99.1\% --- a
2.3-point gap. That gap often determines whether the model meets the
production threshold.''

% -- WARN: Students optimize for theoretical metrics instead of measured performance
Common error: reporting FLOP reduction as the primary success metric.
Correct framing: FLOPs predict performance only for compute-bound workloads.
Most inference is memory-bound, so FLOP reduction alone is misleading.

% -- FLEX: [CORE] These pitfalls connect directly to Ch12 (Benchmarking)
[CORE] The ``profile on target hardware'' message bridges to the next chapters.
IF AHEAD: ``What tools would you use to profile on-device?''
IF SHORT: Cover just the first pitfall (FLOPs vs latency).
}

\small
\textbf{Pitfall:} \textit{Optimizing for theoretical metrics rather than deployment performance.}\\
{\footnotesize Teams reduce FLOPs by 60\% but achieve only 12\% latency reduction on ARM. Memory bandwidth, cache behavior, and instruction-level parallelism determine actual performance --- not operation counts. \textbf{Profile on target hardware.}}

\vspace{0.2cm}
\textbf{Pitfall:} \textit{Using post-training optimization without considering training-aware alternatives.}\\
{\footnotesize PTQ achieves 96.8\% of BERT baseline. QAT retains 99.1\% --- a 2.3-point gap. Post-training pruning to 70\%: 73.8\% accuracy. Gradual pruning \emph{during} training: 75.6\%. The 2--4\% improvement from training-aware methods often determines whether models meet production thresholds.}

\end{frame}

Walk through each bullet. Emphasize the quantitative anchors students
should remember: ``4x from INT8. 16x combined. For LLMs, weight-only INT4

% -- FLEX: [CORE] This is the definitive summary --- read every bullet

\footnotesize
\begin{itemize}\setlength\itemsep{0pt}
  \item \textbf{Three-layer optimization stack}: Structural (what to compute), Precision (how precisely), Architectural (how efficiently). Combine all three.
  \item \textbf{PTQ is a strong baseline}: INT8 post-training quantization delivers 4$\times$ compression with no retraining. Use QAT only when insufficient.
  \item \textbf{For LLMs, weight-only quantization wins}: INT4 weights + FP16 activations dominate because generation is memory-bandwidth bound.
  \item \textbf{Structured pruning for commodity hardware}: Unstructured sparsity needs sparse kernels. Structured pruning (channels, heads) delivers real latency gains.
  \item \textbf{Order matters}: Fuse first (free), quantize second (cheap), prune third (requires tuning), distill last (requires training).
  \item \textbf{Profile on target hardware}: FLOPs and parameter counts mispredict real-world performance. A 2$\times$ FLOP reduction may yield only 1.2$\times$ speedup.
\end{itemize}
