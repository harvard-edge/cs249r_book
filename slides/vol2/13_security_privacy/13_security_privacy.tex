\begin{frame}{Security vs.\ Privacy: Distinct Concerns}
\note{
% -- LINK: What prior concept connects to this slide
The previous slide showed that ML creates new attack surfaces. Now we split
those threats into two distinct categories: security and privacy.

% -- NARRATE: What to SAY while showing this slide
[3 min] Point to the left half of the diagram: ``Security defends against adversarial
manipulation --- someone trying to break your model.'' Point to the right half:
``Privacy defends against unauthorized inference --- someone trying to learn from
your model's outputs.'' Key reveal: ``These can be in tension. Robust models
trained with adversarial examples are MORE susceptible to membership inference
because they memorize more of the training distribution.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``Can you have perfect security without privacy? Privacy without security?''
Give 20 seconds. [Expected: students realize you can lock down a system but still
leak data through the model's outputs.]

% -- WARN: What students will get wrong on THIS topic
Common error: students conflate security and privacy as the same problem.
Correct framing: security requires worst-case analysis (adversarial); privacy
requires statistical guarantees (epsilon-delta). Different math, different tools.

% -- FLEX: [CORE] or [OPTIONAL] + contingency
[CORE] This distinction structures the rest of the chapter.
IF AHEAD: ``Name a real system where security and privacy are in tension.''
IF SHORT: Focus on the callout box at the bottom; skip the full diagram walkthrough.
}

\begin{frame}{Data Poisoning: Extreme Leverage}
\note{
% -- LINK: What prior concept connects to this slide
Model theft targets inference. Data poisoning targets the opposite end of the
lifecycle --- corrupting training data to embed persistent vulnerabilities.

% -- NARRATE: What to SAY while showing this slide
[3 min] Point to the left column (diagram): ``0.1\% corruption --- just 1,000
samples in 1 million --- degrades accuracy 10--50\%.'' Point to the right column:
``Backdoor attacks need only 0.01\% to achieve 90\%+ attack success while
preserving 95\%+ clean accuracy. Backdoors survive retraining and transfer learning.''
ANALOGY: ``Poisoning is like salting a city's water supply --- a tiny amount
contaminates everything downstream.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``How would you detect 1,000 poisoned samples in 1 million?'' Give 15 seconds.
[Expected: students suggest manual review; correct answer is statistical outlier
detection and activation clustering.]

% -- WARN: What students will get wrong on THIS topic
Common error: students think only large-scale corruption matters. Correct framing:
backdoor attacks are effective at 0.01\% precisely because they are invisible to
aggregate accuracy metrics --- the model looks healthy on clean data.

% -- FLEX: [CORE] or [OPTIONAL] + contingency
[CORE] Data poisoning is the highest-leverage attack in the chapter.
IF AHEAD: ``What makes backdoor attacks survive transfer learning?''
IF SHORT: Focus on the 0.1\% and 0.01\% numbers in the right column only.
}

\begin{frame}{Adversarial Examples \& Prompt Injection}
\note{
% -- LINK: What prior concept connects to this slide
Poisoning attacks the training pipeline. Adversarial examples and prompt injection
attack the inference pipeline --- the two runtime-side threats.

% -- NARRATE: What to SAY while showing this slide

misclassification. These transfer across architectures with 60--80\% success ---

craft on a surrogate, apply to production.

\begin{frame}{LLM Prompt Injection: A New Attack Class}
\note{
% -- LINK: What prior concept connects to this slide
Adversarial examples exploit vision models. Prompt injection exploits LLMs ---
a fundamentally different attack surface unique to generative AI.

% -- NARRATE: What to SAY while showing this slide
[3 min] Walk through the three attack tiers: ``Direct injection: `ignore
previous instructions' overrides the system prompt. Indirect injection:
poison documents in a RAG pipeline so the model executes attacker commands
when it retrieves them. Multi-step: chain injections across tool calls in
agentic systems.'' Key punchline: ``There is no buffer overflow here --- the
model cannot distinguish instruction from content because both are text.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``Why is prompt injection fundamentally harder to defend than SQL injection?''
Give 15 seconds. [Expected: SQL injection has a clear boundary between code and
data; LLMs process both in the same token stream.]

% -- WARN: What students will get wrong on THIS topic
Common error: students think input filtering (blocklists) solves prompt injection.
Correct framing: adversaries rephrase continuously. No static filter catches
all injections because the attack surface is the entire natural language space.

% -- FLEX: [CORE] + contingency
[CORE] Prompt injection is the highest-profile LLM vulnerability in production.
IF AHEAD: ``How would you defend a RAG pipeline against indirect injection?''
IF SHORT: Focus on the three-tier taxonomy and the ``no boundary'' insight.
}

\begin{frame}{War Story: Tesla Autopilot IP Theft}
\note{

competitor. The trade secret --- neural network architectures and training

the Economic Espionage Act.'' Key insight: ``Model IP theft does not
require API extraction. Insider access bypasses all external defenses.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``What organizational controls could have prevented this?''
Give 10 seconds. [Expected: access controls, DLP, code watermarking,
exit interviews with forensic audit.]

% -- WARN: What students will get wrong on THIS topic
Common error: students focus entirely on external threats (API extraction).
Correct framing: insider threats account for a significant fraction of
IP theft --- access control and audit trails are as important as API defenses.

\begin{frame}{Differential Privacy: Calibrated Noise}
\note{
% -- LINK: What prior concept connects to this slide
Hardware primitives defend the infrastructure. Differential privacy defends the
data --- it is the mathematical foundation for privacy guarantees.

% -- NARRATE: What to SAY while showing this slide
[3 min] Walk through the diagram: ``DP adds calibrated noise so no individual's
data can be reverse-engineered from the output.'' Point to the three parameters:
``Epsilon is the privacy budget. Sensitivity is how much one record changes the
output. Noise scale = sensitivity / epsilon.'' Key reveal: ``Epsilon=0.1 gives
strong privacy but 10--15\% accuracy loss. This is a quantitative trade-off,
not a toggle.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide

[Expected answer: error per person is approximately Noise/N, so larger N dampens
the per-person noise to acceptable levels.]

% -- WARN: What students will get wrong on THIS topic
Common error: students think DP makes data ``anonymous.'' Correct framing: DP
provides a mathematical bound on information leakage per individual, calibrated
by epsilon. It does not remove individuals from the dataset.

% -- FLEX: [CORE] or [OPTIONAL] + contingency
[CORE] DP is the central privacy mechanism --- all subsequent slides build on it.
IF AHEAD: ``What happens when epsilon approaches infinity?'' [Answer: no privacy.]
IF SHORT: State the epsilon trade-off and skip the diagram walkthrough.

\begin{frame}{Fallacies}
\note{
% -- NARRATE: What to SAY while showing this slide
[2 min] Read each fallacy and its rebuttal aloud. Emphasize the quantitative
counter-evidence: 10K--100K queries for extraction, epsilon compounding to 10x,
70--90\% membership inference on FL, 0.1\% poisoning leverage. The key fallacy
is ``security through obscurity'' --- black-box extraction demolishes it.

% -- FLEX: [CORE] or [OPTIONAL] + contingency
[CORE] Fallacies correct the most dangerous misconceptions from the chapter.
IF SHORT: Read fallacies 1 and 4 only (obscurity and poisoning).
}

\scriptsize
\textbf{Fallacy:} \textit{Security through obscurity protects ML models.}\\
Black-box extraction reconstructs 90\% accuracy with 10K--100K queries, within typical API rate limits.

\vspace{0.06cm}
\textbf{Fallacy:} \textit{Differential privacy automatically ensures privacy.}\\
Budgets compound: 10 runs at $\epsilon=1$ each $=$ total $\epsilon=10$. Organizations exceed guarantees by orders of magnitude.

\vspace{0.06cm}
\textbf{Fallacy:} \textit{Federated learning inherently provides privacy.}\\
Membership inference: 70--90\% accuracy on FL models. Gradient inversion reconstructs training data.

\vspace{0.06cm}
\textbf{Fallacy:} \textit{Data poisoning requires compromising large portions of data.}\\
0.1\% (1K in 1M) reduces accuracy 10--50\%. Backdoors need only 0.01\% for 90\%+ attack success.

\end{frame}

\begin{frame}{Pitfalls}
\note{
% -- NARRATE: What to SAY while showing this slide
[2 min] Read each pitfall. The first is the key takeaway: ``Security is a system
property, not a component property --- strong API defenses are bypassed through
batch endpoints, monitoring logs, or public registries.'' The second connects
to the distributed attack surface slide: O(n-squared) growth.

% -- FLEX: [OPTIONAL] or [CORE] + contingency
[CORE] Pitfalls reinforce the systems-level thinking from the chapter.
IF SHORT: Cover only the first pitfall (component vs.\ system property).
}

\footnotesize
\textbf{Pitfall:} \textit{Treating security as an isolated component.}\\
Strong API defenses are bypassed through batch endpoints, monitoring logs, or public registries. A single weak link compromises the entire system. Security is a \textbf{system property}, not a component property.

\vspace{0.15cm}
\textbf{Pitfall:} \textit{Underestimating distributed attack surface expansion.}\\
The shift from 1 to $n$ machines increases attack vectors by $O(n^2)$, not $O(n)$. Inter-node channels, federated aggregation, and edge deployment each create new vulnerability classes requiring dedicated threat modeling.

\end{frame}

\begin{frame}{Key Takeaways}
\note{
% -- NARRATE: What to SAY while showing this slide
[2 min] Reveal after retrieval practice. Walk through each bullet, anchoring on
the quantitative numbers: 100,000x extraction asymmetry, 0.1\% poisoning leverage,
15\% isolation tax, epsilon compounding. Ask: ``How many of these did you recall?''

% -- FLEX: [CORE] or [OPTIONAL] + contingency
[CORE] This is the synthesis slide --- do not skip.
IF SHORT: Read bullets 2, 3, and 6 (theft, poisoning, layered defense).
}

\scriptsize
\begin{itemize}\setlength\itemsep{1pt}
  \item \textbf{Security $\neq$ Privacy}: Security defends against \emph{adversarial manipulation}; Privacy protects against \emph{unauthorized inference}. Both required.
  \item \textbf{Model theft is cheap}: 100,000$\times$ cost asymmetry --- extraction via 10K--100K queries vs.\ \$100M training.
  \item \textbf{Data poisoning has extreme leverage}: 0.1\% corruption $\to$ 10--50\% accuracy drop; backdoors at 0.01\% persist through retraining.
  \item \textbf{Differential privacy is the standard}: $\epsilon$-$\delta$ guarantees beat anonymization, but budgets compound --- track cumulative $\epsilon$.
  \item \textbf{Hardware is the root of trust}: TEEs (5--30\% cost), Secure Boot, HSMs, PUFs anchor software protections.
  \item \textbf{Layered defense}: No single mechanism suffices. Composition creates graceful degradation.
  \item \textbf{Distributed systems multiply the surface}: Attack vectors grow $O(n^2)$, not $O(n)$.
\end{itemize}

\end{frame}

\footnotesize
\textbf{DP-SGD Implementation Steps:}

\vspace{0.1cm}
\begin{enumerate}\setlength\itemsep{1pt}
  \item \textbf{Set privacy budget}: Choose $\epsilon$ based on regulatory requirements (GDPR: $\epsilon \leq 1$ recommended)
  \item \textbf{Compute sensitivity}: $S = \max \|\nabla L(x_i)\|_2$ after clipping
  \item \textbf{Clip per-sample gradients}: $\bar{g}_i = g_i \cdot \min(1, C / \|g_i\|_2)$
  \item \textbf{Add Gaussian noise}: $\tilde{g} = \frac{1}{B}\sum \bar{g}_i + \mathcal{N}(0, \sigma^2 C^2 I)$
  \item \textbf{Track cumulative budget}: Use R\'enyi DP accountant (tighter than basic composition)
  \item \textbf{Stop when exhausted}: Do not exceed total $\epsilon$ --- including hyperparameter tuning
\end{enumerate}
