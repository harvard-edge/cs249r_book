\section{Fleet Architecture}
% =============================================================================

\begin{frame}{The Machine Learning Fleet}
\note{
% -- LINK: Three walls established the problem; now we name the solution architecture
Students saw failure, bandwidth, and energy walls. The Fleet is the engineering
response --- a warehouse-scale computer, not a cluster of independent servers.

% -- NARRATE: What to SAY while showing this slide
Read the definition aloud. Then walk through the three properties:
``Iterative Statefulness --- the model weights are shared state updated millions of times.
Barrier Synchronization --- 10,000 GPUs wait for the slowest one every step.
Bisection Bandwidth Dominance --- gigabytes cross the network every second.''
ANALOGY: ``Think of an orchestra where every musician must play in perfect sync.
One late violin delays the entire symphony.''

% -- ENGAGE: Specific question for THIS slide
``What happens if one GPU is 10\% slower than the others in BSP training?''
Expected answer: the entire cluster slows by 10\%.

% -- WARN: What students will get wrong
Students conflate ``cluster'' with ``fleet.'' A web cluster handles independent requests;
an ML fleet is tightly coupled. The distinction is barrier synchronization.

% -- FLEX: [CORE] The Fleet definition is the conceptual anchor for Vol II.
IF SHORT: Read the definition, name the three properties, move on.
}

\small
\begin{mlsyscard}{crimson}
\textbf{Machine Learning Fleet:} A distributed system of thousands of interconnected
accelerators, storage arrays, and network fabrics designed to operate as a \emph{single
coherent computer}.
\end{mlsyscard}

\vspace{0.15cm}
\textbf{Three defining properties:}
\begin{enumerate}\setlength\itemsep{0pt}
  \item \textbf{Iterative Statefulness}: Updates a massive shared state (weights) millions of times
  \item \textbf{Barrier Synchronization}: 10,000 GPUs wait for the slowest worker every step
  \item \textbf{Bisection Bandwidth Dominance}: Gigabytes of gradients cross the network every second
\end{enumerate}

\vspace{0.1cm}
\mlsysalert{Key:}{The datacenter \emph{is} the computer. Network = system bus. Orchestrator = OS.}

\end{frame}

\begin{frame}{MapReduce vs.\ ML Fleet}
\note{
% -- LINK: Fleet definition just introduced barrier synchronization
This diagram makes the contrast concrete: MapReduce vs. ML Fleet.

% -- NARRATE: What to SAY while showing this slide
Point to the left side: ``MapReduce: independent partitions, fail in isolation,
retry one shard.'' Point to the right: ``ML Fleet: barrier every step.
A 10\% straggler slows the ENTIRE cluster by 10\%.''
``This is why you cannot borrow MapReduce fault-tolerance strategies.''

% -- WARN: What students will get wrong
Students with Spark/Hadoop experience assume stragglers can be mitigated
by speculative execution. In BSP training, speculation is impossible ---
all replicas must agree on the same gradient.

\begin{frame}{The Reliability Collapse}
\note{
% -- LINK: The GPT-4 MTBF slide gave a single number (4.4 hrs); this shows the curve
Students saw one data point. Now they see the exponential decay function
that governs all fleet-scale reliability.

% -- NARRATE: What to SAY while showing this slide
Point to the curve: ``At 99.9\% per-node reliability, 1,000 GPUs gives you
only 36.8\% fleet availability. At 10,000 GPUs, essentially zero percent.
The x-axis is GPU count; the y-axis is the probability that EVERY GPU is healthy.''
Pause. ``This is why the engineering lesson is: stop preventing failure.
Engineer for resilience --- recovery speed matters more than uptime.''

% -- ENGAGE: Specific question for THIS slide
Ask: ``What does this mean for recovery strategy?''
Expected answer: checkpointing, fast restart, redundancy --- not preventing failure.

% -- WARN: What students will get wrong
Students want to improve per-node reliability (99.99\% instead of 99.9\%).
Show them: $(0.9999)^{10{,}000} = 36.8\%$ --- the same cliff, just shifted right.
The exponential always wins.

% -- FLEX: [CORE] This diagram is the emotional turning point toward fault tolerance.
IF AHEAD: ``At what fleet size does 99.99\% per-node give you 90\% fleet health?''
}

\begin{frame}{Communication Becomes Dominant}

\section{Scaling Laws}

\begin{frame}{AI Scaling Laws}
\note{
% -- LINK: Fleet Architecture showed WHAT we build; Scaling Laws show WHY we must
The fleet exists because scaling laws demand exponentially more compute
for each incremental quality improvement.

% -- NARRATE: What to SAY while showing this slide
Point to the diagram: ``Loss follows a power law: $L(X) \propto X^{-\alpha}$.
Each 10x quality improvement demands 100--1000x more resources.
This is the Bitter Lesson formalized: scale beats cleverness.
And scale means fleets.''

% -- ENGAGE: Specific question for THIS slide
``If you need 10x better performance and scaling laws hold, how many more
GPUs do you need?'' Expected: 100--1000x, not 10x.

% -- WARN: What students will get wrong
Students assume linear returns to compute. The power-law exponent ($\alpha \approx 0.05$--$0.1$)
means massive compute for modest loss reduction.

% -- FLEX: [CORE] Scaling laws justify the entire fleet investment.
IF SHORT: State the power law, give the 10x/100x number, move on.
}

\centering

\vspace{0.15cm}

\begin{frame}{Compute-Optimal Scaling (Chinchilla)}
\note{
% -- LINK: General scaling laws showed WHY scale matters; Chinchilla shows HOW to allocate
Students saw the power law. Chinchilla adds a constraint: there is an optimal
ratio between model size and dataset size for any fixed compute budget.

% -- NARRATE: What to SAY while showing this slide
Point to the equation: ``$D \propto N^{0.74}$. If you double the model size,
you need roughly 1.7x more data to stay compute-optimal.''
Walk through the table: ``GPT-1 to GPT-4: 10,000x more parameters,
10 million times more FLOPs. Each row is a new infrastructure generation.''

% -- ENGAGE: Specific question for THIS slide
Ask: ``What happens if you double parameters but keep the dataset constant?''
Expected: you waste compute --- the model is over-parameterized for the data.

% -- WARN: What students will get wrong
Students assume bigger model = better, regardless of data. Chinchilla showed
that many models (including the original GPT-3) were undertrained --- too large
for their data budget.

% -- FLEX: [CORE] Chinchilla optimality is the quantitative basis for compute budgeting.
IF AHEAD: ``Why might practitioners intentionally train beyond Chinchilla-optimal?''
(Answer: inference cost --- smaller over-trained models are cheaper to serve.)
}

\small
\begin{columns}[T]
  \begin{column}{0.55\textwidth}
    \textbf{Chinchilla optimality} (Hoffmann et al., 2022):

    \vspace{0.1cm}
    $$D \propto N^{0.74}$$

    {\footnotesize Dataset size ($D$) and model size ($N$) must grow
    in coordinated proportions.}

    \vspace{0.15cm}
    \textbf{Three resource regimes:}
    \begin{itemize}\setlength\itemsep{0pt}
      \item {\footnotesize \textcolor{computestroke}{\textbf{Compute-limited}}: Train smaller models longer}
      \item {\footnotesize \textcolor{datastroke}{\textbf{Data-limited}}: Train larger models fewer steps}
      \item {\footnotesize \textcolor{routingstroke}{\textbf{Optimal}}: Chinchilla frontier}
    \end{itemize}
  \end{column}
  \begin{column}{0.42\textwidth}
    \vspace{0.1cm}
    \renewcommand{\arraystretch}{1.15}
    {\footnotesize
    \begin{tabular}{@{}lrl@{}}
      \toprule
      \textbf{Model} & \textbf{Params} & \textbf{FLOPs} \\
      \midrule
      GPT-1  & 117M    & $\sim$$10^{18}$ \\
      GPT-2  & 1.5B    & $\sim$$10^{19}$ \\
      GPT-3  & 175B    & $\sim$$10^{23}$ \\
      GPT-4  & $\sim$1T & $\sim$$10^{25}$ \\
      \bottomrule
    \end{tabular}

\begin{frame}{Fallacies}
\note{
% -- NARRATE: What to SAY while showing this slide
``These are four claims that sound reasonable but are quantitatively wrong.''
Read each fallacy aloud, then deliver the rebuttal number.
Spend extra time on the sparsity fallacy --- it surprises the most students.

% -- FLEX: [CORE] Fallacies correct misconceptions before they solidify.
IF SHORT: Cover the first two fallacies, skip the last two.
}

\footnotesize
\textbf{Fallacy:} \textit{Focusing on FLOPs while ignoring hardware alignment.}\\
{\scriptsize Unstructured pruning: 80\% sparsity but \textbf{no speedup} on dense Tensor Cores. Structured pruning at 50\% guarantees 2$\times$.}

\vspace{0.08cm}
\textbf{Fallacy:} \textit{Efficiency optimizations are ``free wins.''}\\
{\scriptsize INT8 quantization: 4$\times$ memory reduction but 1--2\% accuracy loss. Distillation: 2--4$\times$ compression but expensive teacher.}

\vspace{0.08cm}
\textbf{Fallacy:} \textit{Edge requirements are just ``Cloud Lite.''}\\
{\scriptsize At 120 km/h, every 100 ms delay = 3.33 m of positional uncertainty. Qualitatively different constraints.}

\vspace{0.08cm}
\textbf{Fallacy:} \textit{Scaling laws predict linearly across all scales.}\\
{\scriptsize At 100B+ params, coordination overhead consumes 40\% of compute. 2--3$\times$ cost overruns beyond validated thresholds.}

\end{frame}

\footnotesize
\textbf{Pitfall:} \textit{Extrapolating single-machine intuition to fleet scale.}\\
{\scriptsize Techniques that work for 8 GPUs fail at 8,000 due to the Bisection Bandwidth Wall. Communication overhead grows non-linearly.}

\vspace{0.1cm}
\textbf{Pitfall:} \textit{Treating the network as ``plumbing.''}\\
{\scriptsize Network topology determines throughput more than compute. IB vs.\ Ethernet: 81\% vs.\ 46\% scaling efficiency.}

\vspace{0.1cm}
\textbf{Pitfall:} \textit{Ignoring governance as ``someone else's problem.''}\\
{\scriptsize When a system serves billions, a technical bug becomes a societal risk. Security and fairness from day one.}

\end{frame}

\scriptsize
\begin{itemize}\setlength\itemsep{0pt}
  \item \textbf{Scale creates qualitative change}: $10^7\!\times$ compute growth creates three walls (Network, Reliability, Energy).

  \item \textbf{Failure is the common case}: At 25K GPUs, MTBF $\approx$ 4.4 hours. Engineer for resilience.

about the limits of Chinchilla or Kaplan scaling.
% -- FLEX: [OPTIONAL] Only show if requested.

\footnotesize
\textbf{When do scaling laws stop predicting?}

\vspace{0.15cm}
\begin{itemize}\setlength\itemsep{2pt}
  \item \textbf{Data exhaustion}: High-quality English text estimated at 15--30T tokens. Beyond this, synthetic data introduces distribution shift with unknown scaling properties.
  \item \textbf{Coordination overhead}: At 25K+ GPUs, AllReduce + checkpointing + straggler mitigation consume 40\%+ of total time. MFU drops below useful thresholds.
  \item \textbf{Architecture dependence}: Scaling laws were fit on Transformers. MoE, state-space models, and future architectures may have different exponents.
  \item \textbf{Post-training dominance}: For many applications, RLHF and test-time compute yield better ROI than pre-training scale.
\end{itemize}
