\begin{frame}{Learning Objectives}
\note{

% -- WARN: What students will get wrong on THIS topic

different engineering discipline: thermal throttling, intermittent
connectivity, battery budgets, and device heterogeneity spanning 6+
orders of magnitude.

IF SHORT: Read objectives quickly.
}

\footnotesize
\begin{enumerate}\setlength\itemsep{0pt}
  \item Contrast \textbf{cloud training}, \textbf{on-device learning}, and \textbf{federated learning} by data flow and privacy
  \item Quantify edge training overhead: \textbf{memory} (3--5$\times$), compute (2--3$\times$), energy (10--50$\times$)
  \item Select \textbf{adaptation strategies} (weight freezing, LoRA, sparse) matched to device constraints
  \item Apply \textbf{data efficiency techniques} (few-shot, replay) for limited local data
  \item Design \textbf{federated learning} systems with privacy-preserving aggregation across heterogeneous fleets
  \item Integrate on-device learning into production MLOps with distributed monitoring
\end{enumerate}

\section{Design Constraints}
% =============================================================================

\begin{frame}{Training is Not Inference}
\note{
% -- LINK: What prior concept connects to this slide
The applications showed where edge learning is needed. This slide
reveals why it is hard: training amplifies every constraint.

% -- NARRATE: What to SAY while showing this slide
Walk through the table column by column: ``Inference needs weights plus
one activation map. Training needs weights plus ALL activations (for
backprop) plus gradients plus optimizer state. Memory amplification:
3--5x. Compute: 2--3x (forward + backward + update). Bandwidth: 5--10x
(bidirectional gradient flow). Energy: 10--50x (multiple gradient steps
vs. one forward pass).''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``Why is energy amplification (10--50x) so much worse than memory
amplification (3--5x)?'' Expected: training requires many gradient
steps per adaptation, each consuming energy. Memory is a one-time
allocation; energy compounds with iterations.

% -- WARN: What students will get wrong on THIS topic
Students assume on-device training is just ``inference plus a backward
pass.'' The optimizer state alone (Adam m and v) doubles the memory
footprint beyond weights + activations + gradients.

% -- FLEX: [CORE] or [OPTIONAL] + contingency
[CORE] The amplification table is the central quantitative insight.
IF SHORT: Focus on the 3--5x memory and 10--50x energy numbers.
}

% --- Layout: FULL-WIDTH table ---
\small
\textbf{On-device training amplifies inference constraints multiplicatively:}

\vspace{0.15cm}
\renewcommand{\arraystretch}{1.15}
{\scriptsize
\begin{tabular}{@{}lp{2.8cm}p{3.5cm}p{2.8cm}@{}}
  \toprule
  \textbf{Dimension} & \textbf{Inference} & \textbf{Training} & \textbf{Amplification} \\
  \midrule
  \textbf{Memory} & Weights + 1 activation map & Weights + activations + gradients + optimizer & \textcolor{errorstroke}{\textbf{3--5$\times$}} \\
  \textbf{Compute} & Forward pass only & Forward + backward + update & \textcolor{errorstroke}{\textbf{2--3$\times$}} \\
  \textbf{Bandwidth} & Sequential weight reads & Bidirectional gradient flow & \textcolor{errorstroke}{\textbf{5--10$\times$}} \\
  \textbf{Energy} & Single inference & Multiple gradient steps & \textcolor{errorstroke}{\textbf{10--50$\times$}} \\
  \bottomrule
\end{tabular}
}

\vspace{0.15cm}
\mlsysalert{Consequence:}{A smartphone keyboard uses 25\% of available memory for a \emph{single} training step. Compression is not optional --- it is a survival requirement.}

\end{frame}

\begin{frame}{Three Adaptation Strategies}
\note{
% -- LINK: What prior concept connects to this slide

\begin{frame}{Weight Freezing: Bias-Only Adaptation}
\note{
% -- LINK: What prior concept connects to this slide
The three strategies overview introduced bias-only as the lightest.
This slide shows why it works and how simple the implementation is.

% -- NARRATE: What to SAY while showing this slide
``Standard layer: y = Wx + b. In bias-only adaptation, the gradient
with respect to W is zero --- we never compute or store it. Only the
bias gradient is non-zero. This gives 100--1000x parameter reduction.
Four lines of PyTorch code.'' Point to the code block on the right.

% -- WARN: What students will get wrong on THIS topic
Students dismiss bias-only as ``too simple to work.'' For tasks like
domain adaptation (shifting input distribution), biases capture the
shift surprisingly well because they adjust the decision boundary
without changing feature extraction.

% -- FLEX: [OPTIONAL] + contingency
[OPTIONAL] Implementation detail for the lightest strategy.
IF SHORT: State the 100--1000x reduction and the equation; skip code.
}

\footnotesize
\begin{columns}[T]
  \begin{column}{0.50\textwidth}
    \textbf{Standard layer:} $y = Wx + b$

    \vspace{0.1cm}
    In bias-only adaptation:
    $$\frac{\partial \mathcal{L}}{\partial W} = 0, \quad \frac{\partial \mathcal{L}}{\partial b} \neq 0$$

    \vspace{-0.1cm}
    \begin{mlsyscard}{datastroke}
    {\scriptsize \textbf{100--1000$\times$} parameter reduction. Enables training on devices without FP units.}
    \end{mlsyscard}
  \end{column}
  \begin{column}{0.47\textwidth}
    \begin{mlsyscard}{computestroke}
    {\scriptsize\texttt{%
    \textcolor{midgray}{\# Freeze all parameters}\\
    for p in model.parameters():\\
    \hspace{1em}p.requires\_grad = False\\[0.1cm]
    \textcolor{midgray}{\# Enable bias gradients only}\\
    for name, p in model.params():\\
    \hspace{1em}if "bias" in name:\\
    \hspace{2em}p.requires\_grad = True%
    }}
    \end{mlsyscard}
    {\scriptsize\textcolor{midgray}{PyTorch: 4 lines for 1000$\times$ param reduction.}}
  \end{column}
\end{columns}

\end{frame}

\begin{frame}{LoRA: Low-Rank Adaptation}
\note{
% -- LINK: What prior concept connects to this slide
Bias-only is the lightest strategy. LoRA provides more expressive
adaptation for devices with more resources (smartphones).

% -- NARRATE: What to SAY while showing this slide
``LoRA decomposes weight updates as a low-rank product: W' = W0 + AB,
where A is d-by-r and B is r-by-d, with r much less than d. Storage
per context: 200x smaller than full fine-tuning. Ten user contexts
fit in 42 MB instead of 400 MB.'' Point to the cards: ``Why LoRA works:
weight updates have low intrinsic rank. Rank 4 captures most adaptation
signal. Trade-off: low rank limits expressiveness for complex domain
shifts.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide
Ask: ``Why does rank 4 capture most of the adaptation signal?''
Expected: the weight update has low intrinsic dimensionality --- most
of the change lies in a few principal directions.

% -- WARN: What students will get wrong on THIS topic
Students set rank too high (r=64 or r=128) ``to be safe.'' Higher
rank increases memory, compute, and communication cost without
proportional quality gain. Start with r=4 and increase only if needed.

% -- FLEX: [CORE] or [OPTIONAL] + contingency
[CORE] LoRA is the default adaptation strategy for smartphones.
IF SHORT: Show the equation and the 200x per-context saving.
}

\footnotesize
\begin{columns}[T]

\begin{frame}{Sparse Update Strategies}
\note{
% -- LINK: What prior concept connects to this slide
LoRA adapts all targeted layers uniformly. Sparse updates take a more selective approach: only update the layers that matter most for the new task.

% -- NARRATE: What to SAY while showing this slide
``Three sparse update strategies. Layer Selection: freeze all but the last N layers. Works because early layers learn general features that transfer well. Selective neurons: use Fisher Information to identify the most task-relevant neurons and update only those. Structured pruning during adaptation: remove unimportant connections during fine-tuning, simultaneously adapting and compressing.'' Walk through the comparison table: ``Layer selection is simplest but least expressive. Fisher-based is most precise but requires a calibration pass. The memory savings compound with LoRA: use LoRA on the selected layers only.''

% -- ENGAGE: Specific question, prediction, or task for THIS slide

\begin{frame}{Centralized vs.\ Decentralized Training}
\note{

\begin{frame}{Fallacies}
\note{
% -- LINK: What prior concept connects to this slide
The chapter covered edge constraints, adaptation, data efficiency,
federated learning, and production. These fallacies challenge remaining
misconceptions.

% -- NARRATE: What to SAY while showing this slide
Read each fallacy: ``A smartphone cannot replicate datacenter training
--- 3--10x amplification on 1000x less hardware. Federated learning
does not automatically preserve privacy --- gradient inversion attacks
exist. Local adaptation with bad data can DEGRADE performance ---
fallback to baseline is essential.''

% -- FLEX: [CORE] or [OPTIONAL] + contingency
[CORE] Quantitative rebuttals address the three biggest misconceptions.
IF SHORT: Cover only the privacy fallacy.
}

\small
\textbf{Fallacy:} \textit{On-device learning matches cloud-based training capabilities.}\\
{\footnotesize A smartphone with 8 GB RAM and 5 W cannot replicate datacenter systems with TB of memory and kW of power. Training amplifies resource needs 3--10$\times$ over inference.}

\vspace{0.15cm}
\textbf{Fallacy:} \textit{Federated learning automatically preserves privacy.}\\
{\footnotesize Gradient inversion attacks can reconstruct training images from model updates. True privacy requires differential privacy ($\varepsilon = 1$--$8$, 1--5\% accuracy cost) and secure aggregation.}

\vspace{0.15cm}
\textbf{Fallacy:} \textit{Local adaptation always improves over generic models.}\\
{\footnotesize Adaptation with insufficient or noisy data can \emph{degrade} performance. Systems need fallback to known-good baselines when local data is inadequate.}

\end{frame}

\begin{frame}{Pitfalls}
\note{

``First: assuming uniform hardware across the fleet. Edge spans 6+

% -- FLEX: [CORE] or [OPTIONAL] + contingency
[CORE] The two most underestimated challenges in edge ML.
IF SHORT: Cover only the heterogeneity pitfall.

\footnotesize
\textbf{Pitfall:} \textit{Assuming uniform hardware across the deployment fleet.}\\
Edge capabilities span 6+ orders of magnitude: 32 KB to 16 GB RAM, 48 MHz to 3 GHz, 10 $\mu$W to 5 W. A learning algorithm that works on flagship phones may fail catastrophically on IoT devices. Federated systems must handle \textbf{10,000$\times$} performance differences within a single deployment.

\vspace{0.15cm}
\textbf{Pitfall:} \textit{Underestimating orchestration complexity at scale.}\\
Federated systems must handle intermittent connectivity, version skew across thousands of model variants, participation bias (top 10\% of devices contribute to 50\% of rounds), and devices reconnecting after weeks offline. This is not a scaled-down cloud --- it is a fundamentally different engineering discipline.

\end{frame}

\begin{frame}{Key Takeaways}
\note{
% -- LINK: What prior concept connects to this slide

% -- NARRATE: What to SAY while showing this slide
Walk through each bullet: ``Mobile Memory Wall with 65x bandwidth gap.
Training amplifies 3--10x. Three pillars: adaptation, data efficiency,
federated coordination. Heterogeneity spans 6+ orders of magnitude.
Federated learning is a systems problem. Edge is NOT a scaled-down

% -- FLEX: [CORE] or [OPTIONAL] + contingency
[CORE] Summary after retrieval practice.
IF SHORT: Read bullets quickly.
}

\footnotesize
\begin{itemize}\setlength\itemsep{1pt}
  \item \textbf{Mobile Memory Wall}: NPUs provide TOPS, but on-device models are bandwidth-bound. The 30--50$\times$ gap between mobile RAM and HBM makes quantization mandatory.
  \item \textbf{Training $\neq$ Inference}: On-device learning amplifies resource needs 3--10$\times$ due to activations, gradients, and optimizer state.
  \item \textbf{Three pillars}: Model Adaptation (bias/LoRA/sparse), Data Efficiency (few-shot/replay), Federated Coordination (privacy-preserving aggregation).
  \item \textbf{Heterogeneity is the default}: Edge spans 6+ orders of magnitude. Federated systems must handle stragglers, participation bias, and 10,000$\times$ capability differences.
  \item \textbf{Federated learning is a systems problem}: Stragglers, non-IID data, communication constraints, and privacy attacks require protocol-level mitigations.
  \item \textbf{Edge $\neq$ scaled-down cloud}: Thermal throttling, intermittent connectivity, and battery budgets make edge a fundamentally distinct discipline.
\end{itemize}

\end{frame}
