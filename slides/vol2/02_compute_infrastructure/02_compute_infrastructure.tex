\begin{frame}{The Infrastructure Stack}
\note{
% -- LINK: Ch1 introduced the Fleet Stack conceptually; now we zoom into the Infrastructure layer
Students saw Infrastructure at the bottom of the Fleet Stack. This slide
expands that single layer into four physical levels.

% -- NARRATE: What to SAY while showing this slide
Walk bottom-to-top: ``Accelerator: bounded by the Memory Wall --- HBM bandwidth
limits throughput. Node: 8 GPUs connected by NVLink to pool capacity.
Rack: where the Power Wall hits --- 33--40 kW per ML rack. Pod: the Communication
Wall --- InfiniBand binds nodes into a warehouse-scale computer.
Each level exists because a constraint at the level below became intolerable.''

% -- ENGAGE: Specific question for THIS slide
Ask: ``Which wall do you think is hardest to overcome?''
No right answer --- the goal is to prime thinking about interdependencies.

% -- WARN: What students will get wrong
Students pick one wall and treat it as independent. In reality, solving
the Memory Wall (bigger HBM) worsens the Power Wall (more watts per chip).

% -- FLEX: [CORE] This is the organizing framework for the entire chapter.
IF SHORT: Name the four levels and their walls, skip the question.
}

\section{Accelerators}
% =============================================================================

\begin{frame}{The Accelerator Spectrum}
\note{
% -- LINK: Infrastructure Stack showed the accelerator level; now we zoom into accelerator design
Students know the accelerator is the atomic unit. This slide shows the design
space: CPU, GPU, TPU, ASIC --- a trade-off continuum, not a ranking.

% -- NARRATE: What to SAY while showing this slide
Point left to right: ``CPU: most flexible, least efficient for ML. GPU: reclaims
die area from branch prediction and caches, spends it on multipliers. TPU: removes
the GPU's graphics legacy, pure matrix engine. ASIC: fully custom, maximum throughput,
zero flexibility.'' Emphasize: ``The Generality Tax: every transistor spent on
branch prediction is a transistor that could have been a multiplier.''

% -- ENGAGE: Specific question for THIS slide
Ask: ``Why would anyone choose a GPU over a custom ASIC?''
Expected: flexibility --- GPUs run any model; ASICs are locked to one architecture.

% -- WARN: What students will get wrong
Students assume ASICs are always superior. Correct: ASICs take 2--3 years to design.
If the model architecture shifts (it does), the ASIC is obsolete before deployment.

% -- FLEX: [CORE] The Generality Tax is a foundational concept for hardware reasoning.
IF AHEAD: ``What happens to the Generality Tax as model architectures converge on Transformers?''
}

\begin{frame}{GPU Architecture Evolution}
\note{
% -- LINK: Accelerator Spectrum showed the GPU's position; now we trace its evolution
Students placed the GPU on the spectrum. This slide shows four generations
of co-evolution between hardware and ML workloads.

% -- NARRATE: What to SAY while showing this slide
Point to each generation: ``Volta: first Tensor Cores, addressed the compute gap.

Hopper: Transformer Engine, addressed the dominant architecture.
Blackwell: multi-die design, addressed the reticle limit.''
Emphasize: ``Efficiency grew 70x, but raw TFLOPS grew 236x. The 3.4x gap

Expected: memory bandwidth or interconnect --- the walls are shifting.

% -- WARN: What students will get wrong
Students focus on peak TFLOPS as the key metric. Correct: the TFLOPS/W column
reveals the efficiency wall. Raw performance is meaningless if you cannot power it.

% -- FLEX: [CORE] The generational pattern teaches students to think about hardware evolution.
IF SHORT: Point to the first and last rows, state the 236x vs 70x gap, move on.

\section{Memory Wall}
% =============================================================================

\begin{frame}{The Memory Wall}
\note{
% -- LINK: Accelerator section showed compute power; now the Memory Wall shows its limit
Students saw impressive TFLOPS numbers. This slide reveals that those Tensor
Cores sit idle 99\% of the time during inference because data cannot arrive fast enough.

% -- NARRATE: What to SAY while showing this slide
Point to the diagram: ``HBM: 3,350 GB/s --- 50x faster than DDR5. 10x more
energy efficient. But even HBM cannot keep the Tensor Cores fed during single-token
generation. The paradox: faster logic idles MORE, waiting for data.''
Point to the table: ``The 50x bandwidth gap and 10x energy gap are why HBM exists.''

% -- ENGAGE: Specific question for THIS slide
Ask: ``If you could double either FLOPS or bandwidth, which helps more for
serving a chatbot?'' Expected: bandwidth --- inference is memory-bound.

% -- WARN: What students will get wrong
Students assume more TFLOPS helps inference. Correct: at arithmetic intensity of 1,
the workload is deep in the memory-bound regime. More compute yields zero benefit.

% -- FLEX: [CORE] The Memory Wall is the single most important concept for inference.
IF AHEAD: ``What about batched inference? Does batching change the answer?''
(Yes --- higher batch size increases arithmetic intensity.)
}

\begin{frame}{Token Latency: The Physics}
\note{
% -- LINK: Memory Wall introduced the paradox qualitatively; this makes it quantitative
Students heard that logic idles waiting for data. Now they see the exact numbers.

% -- NARRATE: What to SAY while showing this slide
Walk through the table row by row: ``Llama-3 70B on H100. Compute: 2 times 70B
FLOPs divided by 1,979 TFLOPS = 0.07 ms. Memory: 140 GB of weights divided by
3.35 TB/s = 41.8 ms. The processor spends 97\% of its time waiting for data.''
Pause for effect. ``Even a chip with INFINITE compute generates tokens only
negligibly faster. Memory bandwidth is the binding constraint.''

% -- ENGAGE: Specific question for THIS slide
``If HBM4 delivers 6 TB/s (roughly 2x), what is the new token latency?''
Expected: approximately 23 ms --- nearly 2x speedup, confirming bandwidth is the lever.

% -- WARN: What students will get wrong
Students will try to optimize the compute term (use quantization to reduce FLOPs).
Correct: the compute term is already 3\% --- even zeroing it out saves 0.07 ms.

% -- FLEX: [CORE] This worked example is the quantitative proof of the Memory Wall.
IF SHORT: Show the table, state the 97\% number, skip the derivation walkthrough.
}

\section{Roofline}
% =============================================================================

\mlsysfocus{The Roofline Model}{%
$\text{Achievable FLOPS} = \min\left(R_{\text{peak}},\; \text{BW} \times I\right)$\\[0.3cm]
\normalsize where $I = \text{FLOP/byte}$ (arithmetic intensity)%
}

\begin{frame}{Roofline: H100 Landscape}
\note{
% -- LINK: Students just predicted which workload benefits from faster GPUs; this is the reveal
The Roofline Model provides the visual framework that answers the prediction.

% -- NARRATE: What to SAY while showing this slide
``Here is the answer to the prediction. The Roofline has two regimes separated
by the ridge point at 591 FLOP/byte. Left of the ridge: memory-bound --- more
TFLOPS is wasted money. Right of the ridge: compute-bound --- more bandwidth
is wasted money. LLM decode at batch 1: far left, less than 1\% of peak.
LLM training: crosses the ridge. The answer: training benefits from more TFLOPS;
inference benefits from more bandwidth.''

% -- ENGAGE: Specific question for THIS slide
``Where on this plot does YOUR workload sit?'' Ask students to think about
a model they have trained or served.

% -- WARN: What students will get wrong
Students confuse arithmetic intensity with utilization. High AI does not mean
high utilization --- you can be compute-bound and still only achieve 30\% of peak
due to scheduling overhead.

% -- FLEX: [CORE] The Roofline is the diagnostic tool students will use throughout the course.
IF AHEAD: ``What shifts a workload horizontally on this plot?'' (Batch size.)
}

% --- ACTIVE LEARNING 2: Exercise ---
\begin{frame}{Your Turn: Roofline Diagnosis}
\note{
% -- LINK: Roofline table showed workload placement; now students calculate it themselves
Students saw pre-computed placements. Now they derive arithmetic intensity from first principles.

% -- NARRATE: What to SAY while showing this slide
``90 seconds. A 7B model, 14 GB in FP16, served at batch 1 on H100.
Calculate the arithmetic intensity and determine the regime. Work with
your neighbor.''
After pause: ``$I = 2 \times 7B / 14$ GB = 1 FLOP/byte. Ridge point is 591.
1 is 591x below the ridge --- deeply memory-bound. More TFLOPS yields zero
benefit. You need more bandwidth.''

% -- ENGAGE: This IS the active learning moment
After the solution: ``What batch size would push this workload to the ridge?''
Expected: roughly 591 --- but that is impractical for latency-sensitive serving.

% -- WARN: What students will get wrong

\begin{frame}{Fallacies}
\note{
% -- NARRATE: What to SAY while showing this slide
``Three claims that sound reasonable but are quantitatively wrong.''
Spend extra time on the Peak FLOPS fallacy --- it connects directly to the Roofline.
``We just showed that LLM inference uses less than 1\% of peak. Buying a faster
GPU for inference is literally wasting money.''

% -- FLEX: [CORE] Fallacies correct misconceptions before they solidify.
IF SHORT: Cover the first two fallacies, skip the liquid cooling one.
}

\small
\textbf{Fallacy:} \textit{More GPUs always means faster training.}\\
{\footnotesize Communication overhead grows with cluster size. Scaling from 1,024 to 2,048 GPUs doubles cost but may reduce training time by only 30--40\%.}

\vspace{0.15cm}
\textbf{Fallacy:} \textit{Peak FLOPS determines training throughput.}\\
{\footnotesize LLM inference operates at 1--2 FLOP/byte, achieving $<$1\% of peak FLOPS. The Roofline Model's ridge point is the diagnostic that matters, not the spec sheet.}

\vspace{0.15cm}
\textbf{Fallacy:} \textit{Liquid cooling is too expensive.}\\
{\footnotesize Over 3 years, a 1,000-GPU cluster saves \$540K in electricity with liquid cooling --- typically exceeding the incremental CapEx.}

\end{frame}

\begin{frame}{Pitfalls}
\note{
% -- NARRATE: What to SAY while showing this slide
``These are mistakes teams actually make.'' The power/cooling pitfall is the
most common: ``Teams buy GPUs before confirming the facility can handle them.
Result: expensive silicon in shipping containers in the parking lot.''
The arithmetic intensity pitfall connects to the Roofline exercise.

% -- FLEX: [CORE] Pitfalls prevent expensive operational mistakes.
IF SHORT: Cover the power/cooling pitfall only --- it is the most impactful.
}

\footnotesize
\textbf{Pitfall:} \textit{Ignoring power and cooling constraints during planning.}\\
A rack of 4 DGX H100s requires 33--40 kW. Upgrading facility electrical infrastructure takes 12--18 months. GPUs arrive in 6 months. Result: expensive silicon in shipping containers in the parking lot.

\vspace{0.15cm}
\textbf{Pitfall:} \textit{Buying the latest accelerator without checking arithmetic intensity.}\\
B200 (4,500 TF) for 7B inference at B=1: arithmetic intensity $\approx$ 1 FLOP/byte. The B200 runs only 2.4$\times$ faster than H100 despite 2.3$\times$ higher peak FLOPS. You paid for compute you cannot use.

\vspace{0.15cm}
\textbf{Pitfall:} \textit{Treating all GPU-hours as equivalent in TCO analysis.}\\
An H100 GPU-hour delivers 6.3$\times$ the TFLOPS of an A100 hour. The right metric is cost per useful \textbf{TFLOP-hour}, not cost per GPU-hour.

\end{frame}

\begin{frame}{Key Takeaways}
\note{
% -- LINK: Students just wrote their own key ideas; now compare with the official list

% -- NARRATE: What to SAY while showing this slide
``Check your list. Did you capture: 97\% memory time, 18x bandwidth cliff,

calibrate your engineering intuition for compute infrastructure.''

% -- FLEX: [CORE] Official summary. Read every bullet.

\scriptsize
\begin{itemize}\setlength\itemsep{0pt}
  \item \textbf{Generality Tax}: CPUs devote 5--10\% of die to math; GPUs 50--60\%; ASICs 90\%+. Flexibility traded for throughput.
  \item \textbf{Memory Wall}: Token generation: 97\% memory transfer, 3\% compute. HBM: 50$\times$ DDR BW, 10$\times$ efficiency.
  \item \textbf{Roofline Model}: Ridge point = $R_{\text{peak}}/\text{BW}$. Below $\to$ bandwidth-bound. Above $\to$ compute-bound. Diagnose before optimizing.
  \item \textbf{Bandwidth Hierarchy}: 18$\times$ cliff (NVLink $\to$ IB) dictates parallelism. Tensor parallel within node; data parallel across.
  \item \textbf{Power \& Cooling}: ML racks: 33--40 kW (10$\times$ traditional). Liquid PUE 1.05 vs.\ air PUE 1.4--1.6.
  \item \textbf{TCO}: Build-vs-buy break-even at $\sim$55\% utilization. Metric: cost/TFLOP-hour, not cost/GPU-hour.
  \item \textbf{Stack}: Each level's constraint creates the next level. No component optimized in isolation.
\end{itemize}

\end{frame}
