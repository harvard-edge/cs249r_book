\chapter{Data Foundations}\label{data-foundations}

This appendix covers the principles of data engineering and statistical
monitoring that keep ML systems healthy. From storage formats that
determine I/O throughput to drift metrics that detect silent failures,
these foundations connect directly to the data pipelines in
\textbf{?@sec-data-engineering-ml} and the operational monitoring in
\textbf{?@sec-machine-learning-operations-mlops}.

\section{Data Engineering
Foundations}\label{sec-system-foundations-data-engineering-foundations-9ba2}

Understanding hardware constraints is only half the battle; we must also
shape our data to fit them. \textbf{Data Engineering} applies the
principles of the memory hierarchy to storage formats and pipeline
design, ensuring that the GPU never starves.

This process begins with recognizing that data is physical---it has
volume, it takes time to move, and it requires energy to parse.

Unlike compute, which gets faster every year, the speed of light is
fixed and network bandwidth is a finite resource. When we speak of
``Data Gravity,'' we are describing the phenomenon where datasets become
so large that they effectively become stationary.

\subsection{Row vs.~Columnar
Formats}\label{sec-system-foundations-row-vs-columnar-formats-1f97}

The choice of file format determines the ``physics'' of how your data is
read.

\textbf{Row-Oriented (CSV, JSON)}: Data is stored record-by-record. To
read just the \texttt{age} column, you must scan every byte of every
row. This is efficient for \emph{writing} (appending a log) but
inefficient for \emph{analytics} (training on specific features).

\textbf{Column-Oriented (Parquet, Arrow)}: Data is stored
column-by-column. To read the \texttt{age} column, the disk head seeks
to that column's block and reads it sequentially. This enables
\textbf{projection pushdown} (reading only the bytes you need) and
\textbf{vectorized processing} (SIMD operations on columns).
Figure~\ref{fig-row-vs-col} contrasts these two storage layouts.

\begin{figure}[htb]

\centering{

\pandocbounded{\includegraphics[keepaspectratio]{index_files/mediabag/5fac2fc59f34fe437f107a4c363e103cf670e133.pdf}}

}

\caption{\label{fig-row-vs-col}\textbf{Storage Layouts}: Row-oriented
formats pack data together by record (good for transactions).
Column-oriented formats pack data by feature (good for analytics).}

\end{figure}%

This layout difference has direct consequences for ML pipelines where
GPU utilization depends on data loading speed.

\begin{tcolorbox}[enhanced jigsaw, colback=white, left=2mm, bottomrule=.15mm, coltitle=black, colbacktitle=quarto-callout-note-color!10!white, arc=.35mm, bottomtitle=1mm, rightrule=.15mm, opacityback=0, colframe=quarto-callout-note-color-frame, leftrule=.75mm, title=\textcolor{quarto-callout-note-color}{\faInfo}\hspace{0.5em}{Systems Insight: The GPU Starvation Problem}, toptitle=1mm, breakable, titlerule=0mm, opacitybacktitle=0.6, toprule=.15mm]

The choice of file format determines whether your system is \textbf{I/O
Bound} or \textbf{Compute Bound}. In row-oriented formats (CSV/JSON),
the CPU must parse text strings into floating-point numbers---a task
that is often 10--100x slower than the GPU's ability to process those
numbers. Using columnar formats like Parquet or Arrow allows for
\textbf{zero-copy reads} and \textbf{projection pushdown}, ensuring that
the ``Data Movement'' term in the Iron Law does not become a bottleneck
that leaves your expensive GPUs idling.

\end{tcolorbox}

\subsection{The Algebra of
Data}\label{sec-system-foundations-algebra-data-7b3d}

Feature engineering is built on three SQL primitives. Understanding
their computational cost prevents pipeline bottlenecks.

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \textbf{Selection (\(\sigma\))}: Filtering rows (e.g.,
  \texttt{WHERE\ age\ \textgreater{}\ 30}).

  \begin{itemize}
  \tightlist
  \item
    \emph{Cost}: Cheap (\(O(\log N)\)) if indexed; expensive (\(O(N)\))
    if full scan.
  \end{itemize}
\item
  \textbf{Projection (\(\pi\))}: Selecting columns (e.g.,
  \texttt{SELECT\ age}).

  \begin{itemize}
  \tightlist
  \item
    \emph{Cost}: \textbf{Free} in columnar formats (only read relevant
    blocks). In row formats, you must read the entire 1KB row just to
    extract the 4-byte integer, wasting 99\% of I/O bandwidth.
  \end{itemize}
\item
  \textbf{Join (\(\bowtie\))}: Combining tables. The most expensive
  operation.

  \begin{itemize}
  \tightlist
  \item
    \textbf{Shuffle Join}: Both tables are partitioned by key and
    exchanged over the network.

    \begin{itemize}
    \tightlist
    \item
      \emph{Cost}: Massive network traffic. Joining two 1 TB tables
      requires moving \textasciitilde2 TB over the network.
    \end{itemize}
  \item
    \textbf{Broadcast Join}: One small table is sent to all workers.

    \begin{itemize}
    \tightlist
    \item
      \emph{Cost}: Minimal network, but the small table \emph{must} fit
      in RAM.
    \end{itemize}
  \end{itemize}
\end{enumerate}

\section{Probability \& Statistics for
Systems}\label{sec-system-foundations-probability-statistics-systems-a2cb}

Once data is flowing through our pipelines, we need mathematical tools
to ensure its quality and consistency. \textbf{Probability \&
Statistics} provide the language for monitoring system health, detecting
the silent failures of data drift, and managing uncertainty.

\phantomsection\label{callout-perspectiveux2a-1.1}
\begin{fbx}{callout-perspective}{Systems Perspective:}{Why This Matters}
\phantomsection\label{callout-perspective*-1.1}
Your monitoring dashboard says average latency is fine, but users are
complaining. Why? Because systems live in the ``long tail.'' Statistics
gives us the tools to measure uncertainty, detect drift, and handle
numerical stability.

\end{fbx}

\subsection{Distributions and the Long
Tail}\label{sec-system-foundations-distributions-long-tail-37cc}

In systems, the \textbf{mean} is often misleading. Latency distributions
are almost always \textbf{long-tailed} (log-normal or power law). A
``P99'' (99th percentile) latency of 500ms means 1\% of your users
experience extreme slowness, which at scale (1M users) is 10,000 unhappy
people.

\phantomsection\label{callout-notebookux2a-1.2}
\begin{fbx}{callout-notebook}{AI Engineer’s Notebook:}{Napkin Math: The Median Experience}
\phantomsection\label{callout-notebook*-1.2}
If a user session involves \(N=100\) requests (common for a web page
load or chat session), the probability of experiencing \emph{at least
one} P99 latency spike is:
\[ P(\text{Slow}) = 1 - (0.99)^{100} \approx 1 - 0\.366 = 63\.4\% \] For
a heavy user (\(N\) is large), the ``tail'' latency becomes their
\emph{median} experience. This is why Google and Amazon optimize for
P99.9 or P99.99.

\end{fbx}

\subsection{Measuring Drift
(Divergence)}\label{sec-system-foundations-measuring-drift-divergence-0525}

Because distributions have long tails where the most dangerous failures
hide, simple metrics like ``mean shift'' are insufficient. We need tools
that compare the \emph{entire shape} of the distribution.

How do we know if serving data (\(Q\)) has drifted from training data
(\(P\))? We measure the ``distance'' between their distributions.

\textbf{KL Divergence} measures how much information is lost if we
approximate \(P\) with \(Q\)
(\citeproc{ref-kullback1951information}{Kullback and Leibler 1951}):
\[ D_{KL}(P || Q) = \sum P(x) \log \frac{P(x)}{Q(x)} \]

\textbf{Population Stability Index (PSI)} is a symmetric,
industry-standard metric derived from KL divergence. A PSI
\textgreater{} 0.2 typically triggers an alert for retraining.

\subsection{Information Theory for
Systems}\label{sec-system-foundations-information-theory-a8f2}

In the ``Information Roofline'' model (\textbf{?@sec-appendix-dam}), we
treat data quality as a physical constraint similar to bandwidth.
Information Theory provides the units for this constraint.

\begin{itemize}
\tightlist
\item
  \textbf{Entropy (\(H\))}: The average information content
  (uncertainty) in a distribution. A uniform distribution has maximum
  entropy (maximum uncertainty).
\item
  \textbf{Information Density}: The amount of useful signal per unit of
  storage. High-quality data has high information density; noisy data
  has low density.
\item
  \textbf{Signal-to-Noise Ratio (SNR)}: The ratio of useful information
  to irrelevant variance. In ML, training on low-SNR data is like trying
  to learn a pattern from static---you hit the ``Information Roofline''
  where adding more compute yields no improvement.
\end{itemize}

\subsection{Logits and Numerical
Stability}\label{sec-system-foundations-logits-numerical-stability-9f55}

Neural networks output \textbf{logits} (unnormalized scores), not
probabilities. We convert them using Softmax:
\[ \text{Softmax}(z_i) = \frac{e^{z_i}}{\sum e^{z_j}} \]

\textbf{The Problem}: If \(z_i\) is large (e.g., 100), \(e^{100}\)
overflows floating-point range. \textbf{The Solution}: We compute in
\textbf{log-space}. The ``Log-Sum-Exp'' trick allows us to compute
\(\log(\sum e^{z_j})\) without ever calculating the massive exponentials
directly, preserving numerical precision.\sidenote{This is implemented
as \texttt{torch.logsumexp} in PyTorch and
\texttt{scipy.special.logsumexp} in SciPy. It relies on the identity
\(\log(\sum e^{x_i}) = a + \log(\sum e^{x_i - a})\), where
\(a = \max(x_i)\). Shifted values \(x_i - a\) are \(\le 0\), ensuring
exponentials never overflow. }
